## 1. What ResearchOS is

A local-first research project management app for science labs:

- GANTT scheduling with dependency-aware date shifts.
- Lab notes + results: markdown editor with image strip, drag-drop attachments, image gallery picker.
- PCR protocol builder with reagent calculators.
- Methods library with reusable protocols.
- Multi-user shared folders (OneDrive / Dropbox / iCloud) with per-user accounts, password gates, and shared-task editing.
- Telegram bot ingestion тЖТ image inbox.
- External ICS calendar overlays.

**Architecture:** Next.js 16 (App Router) + React 19 + TypeScript + Tailwind. **No backend.** All data lives in a folder on the user's disk, accessed via the **File System Access API**. The only server-side code is two thin proxy routes (`/api/telegram-file`, `/api/calendar-feed`) that exist purely to bypass CORS for those integrations.

**Hosted at:** [research-os.app](https://research-os.app/) (the live public domain, auto-deploys from `origin/main`). Also runs locally via `./start.sh` (frontend only; the old FastAPI backend was deleted).

**Positioning / pricing (UPDATED 2026-06-10, the SOLIDARITY MODEL, supersedes any flat "cost-recovery only" framing).** The local-first app is free and open source forever, **AGPLv3**, funded by a UW Distinguished Research Fellowship at UW-Madison plus voluntary GitHub Sponsors, so it stays free for low-resource labs. The ONLY paid part is **optional cloud storage** above a generous free pool. Pricing is **solidarity-based, NOT flat cost-recovery**: individuals and labs pay cost recovery (what storage actually costs us, no more, on the Free / Plus / Pro and Lab Free/Plus/Pro tiers), while **departments and institutions pay a modest sustaining rate ABOVE bare cost**, and that surplus keeps ResearchOS free for individual researchers and funds the open-source development. Dept and institution are **automated self-serve plan builders** (you build a monthly plan from labs/members/adoption), billed by an **auto recurring Stripe invoice to procurement** (GitHub Sponsors cannot invoice a university, so the invoice IS the institutional channel; Sponsors stays the give-extra path for individuals). ## 4. Working conventions

### Product philosophy (the standing values, read these first)

These are the durable values every agent optimizes for. They override convenience and build-effort savings.

1. **Efficiency is the name of ResearchOS.** The whole product exists to save researchers time and effort. When two solutions are on the table, pick the one that is best for the USER's efficiency and experience, even when it is by far the most work for US to build. Build effort is never a reason to ship a worse experience. The amount of work it takes us is our problem to solve, not the user's. (Canonical example, 2026-06-14: for the reproduce-from-PDF figure-style match we chose the full figure picker with page thumbnails + drag-to-crop over the cheaper auto-send-all-pages shortcut, because a clean cropped figure gives the user a faithful match and zero wrong-figure failures. When a build decision trades user value against our effort, state the tradeoff plainly and default to user value, the way the figure-picker decision did.

### Sub-bot anchor drift (2026-05-27, learned the hard way)

A spawned worktree branches from the commit `main` pointed at **when the agent was launched**, not live `main`. During a long session the orchestrator lands dozens of cherry-picks onto `main`, so by the time a sub-bot finishes, its anchor can be 30+ commits stale. This bit us repeatedly in one night. Two halves to the rule:

**Sub-bot side тАФ rebase before committing.** Brief every spawned bot: "FIRST run `git merge main --no-edit` in your worktree root and confirm it succeeds before doing any work. If it conflicts non-trivially, STOP and report." A bot that rebases onto current `main` produces a commit that cherry-picks cleanly with no conflict resolution at all. The telegram-image-path bot did this and its cherry-pick was a clean fast-forward; the bots that didn't caused the messes below.

**Orchestrator side тАФ never `-X theirs` a stale-anchor commit onto a fast-moving main.** `-X theirs` resolves every conflict by taking the cherry-picked (stale) side. When that side is older than `main`, it **silently reverts** whatever landed on `main` after the anchor. Concrete damage from one night:
- A telegram-tutorial cherry-pick reverted 424 lines of `batch-routing.test.ts` coverage that had landed post-anchor (tests still "passed" тАФ on a shrunken set). Had to restore main's file and re-graft the new tests.
- A workbench-create cherry-pick auto-merged `step-machine.ts` from the stale base and produced **duplicate step ids** + a dropped `settings-tour-calendar` slot, which only surfaced as test failures. Had to revert the whole commit and re-apply the change surgically by hand on top of current main.

**Verification after every cherry-pick:** run `git show <hash> --stat` and sanity-check the line deltas. If a file you didn't expect to shrink lost hundreds of lines (especially a test file), that's anchor-drift clobbering, not an intended edit. Diff the file against `main`'s pre-cherry-pick version (`git show <main-before>:path | wc -l`) and recover the lost content. Better: prefer **merging the sub-bot's branch** over `cherry-pick -X theirs` when the branch was rebased onto main; reserve `-X theirs` for the case where you've already confirmed the only divergence is the intended change.

**Circular-import footgun (same night):** `step-registry тЖТ WikiPointerStep тЖТ TourBootstrap тЖТ step-registry` closed a cycle that Next.js tolerated but vitest's module loader did not тАФ it silently broke the entire `step-bodies.test.tsx` import (0 tests ran), masking every step-body assertion for who-knows-how-long. When a registry imports a step that imports a bootstrap that imports the registry, extract the shared leaf helpers into a dependency-free module (the `wiki-pointer-nav-flag.ts` / `lib/query-client.ts` pattern). If a whole test file reports "no tests" instead of pass/fail, suspect an import cycle, not an empty file.

### Sub-bot worktree discipline (2026-05-29, current hard rules)

The non-negotiable preamble every worktree sub-bot brief must carry, learned across the cohort sessions:

- **Symlink node_modules for the tsc/vitest gate; `pnpm install` only when a live dev server is genuinely needed.** `ln -s /Users/gnickles/Desktop/ResearchOS/frontend/node_modules frontend/node_modules` covers the verify gate (never `npm install` тАФ wrong PM, and a bare install once revoked macOS TCC and killed every agent). But a symlinked/COW node_modules does NOT boot a Turbopack dev server (it escapes the worktree root), so a task that must drive the live app (Preview MCP, browser walkthrough) needs a real `pnpm install --prefer-offline` in its OWN isolated worktree via `scripts/worktree-dev.sh` тАФ that ran cleanly many times on 2026-06-11 with no TCC recurrence. Never install into a worktree that is symlinked to or shared with the main store; isolate it. pnpm is pinned to 10.34.3 (Corepack auto-switches).
- **NEVER `git stash`.** The stash is a single repo-wide resource shared across every worktree; one bot stashing has wiped another session in-progress stash. If a bot needs a clean tree it commits or discards its own exact paths, never stash.
- **Never pipe a gate/test/build command through `tail`** (or any pipe). The pipe exit code masks the real one, so a failing `tsc`/vitest reads as green. Use `cmd > /tmp/x.log 2>&1; echo "EXIT=$?"` then read the log. The login shell here runs with `set -e`, so also guard any command that can legitimately exit non-zero (a `grep` with no match) with `|| true`.
- **Avoid apostrophes in heredoc commit bodies.** Commits use `git commit -m "$(cat <<'EOF' ... EOF)"`; a stray apostrophe inside the quoted heredoc breaks the quoting and produces a garbled or failed commit. Reword to drop it, or use `git commit -F /tmp/msg.txt` for a body that must contain one.
- **Realign onto live main before working** (see anchor-drift above) and **commit one coherent commit on your branch, do not merge or push** тАФ the orchestrator cherry-picks. `git add` only your exact paths; the tree is shared with other agents.

These five are the standard preamble; paste them into every worktree brief.

### Mechanics verifiers: full sequential walk required (2026-05-25)

Seed-jump probing and sequential walks are complementary, not substitutes. Mechanics verifiers MUST do at least one full sequential walk per redesign verification, in addition to any per-step seed-jump probing. Seed-jumps isolate a single step's behavior from a clean room; sequential walks expose state-dependent bugs where a flag or DOM mutation set in an earlier step changes how a later step behaves.

The runbook for every mechanics-verifier brief must include: "Walk sequentially from `<start step>` to `<end step>` at least once before declaring the section verified," alongside any per-step `wizardSeedStep` probing the brief calls for. Treat the sequential pass as the integration test and seed-jumps as the unit tests.

Recent example (┬з6.2b R3, home widgets). The R3 mechanics verifier seed-jumped directly into Step 5 from clean fixture state and reported the onEnter actions as PASS. The R3 fresh-eyes verifier walked sequentially from Step 1 and reported Step 5's onEnter `el.click()` BROKEN: the add-demo action in Step 3 had armed the InputLockOverlay's capture-phase handler, which then swallowed the Step 5 click. Seed-jumping bypassed the Step 3 setup entirely, so the bug was invisible to the mechanics pass.

### Verifiers fix spotlight / highlight-box bugs in-place (2026-05-25)

Standard verifier rule: read-only, file findings, defer fixes to a separate chip. Carve-out: the v4 tour's blue spotlight / highlight box around UI elements. Verifiers with the live Preview MCP session already up should fix those in-place and re-verify in the same session, rather than round-tripping to a separate fix chip.

Why: visual positioning bugs (wrong `data-tour-target` selector, target off-screen, ring sized wrong, dim layer mis-placed, spotlight pointing at parent when it should point at child) require live render to dial in. A separate fix chip has to re-setup the Preview environment, re-walk the tour to the failing step, and iterate without the verifier's existing mental context. In-place fix is strictly less work.

Brief addendum to bake into every walkthrough-verifier brief: "Standard rule: do not modify code. Verifiers still defer fixes that touch step bodies, copy, sequence logic, or anything requiring a design decision.

### Field migrations

When a field on Task / Method / Project / Note / etc. is renamed or restructured, follow the **lazy-normalize + on-demand-repair** pattern the cleanup pass landed (commit `147db270`, 2026-05-13). The whole point is that shared on-disk files from other users with legacy shapes keep working transparently тАФ no flag-day cutovers, no broken receivers.

1. **Rename the field** in `frontend/src/lib/types.ts` and update every caller in lockstep. (See the ┬з6 "Field renames" trap for the grep checklist тАФ partial renames silently typecheck.)
2. **Add a `normalize<Entity>Record` helper** at the read boundary in `frontend/src/lib/local-api.ts`. Templates: `normalizeMethodRecord`, `normalizeTaskRecord`. The helper detects the legacy shape, rewrites it in-memory to the new shape, returns the normalized record. Apply on every read path (`.get`, `.list`, `fetchAllXIncludingShared`, тАж) so callers never see the legacy shape.
3. **Add a one-shot repair button** to `frontend/src/app/settings/page.tsx` under "Data maintenance." It iterates every stored file of that entity, normalizes, writes back. Report scanned / repaired / already-clean counts. Safe to re-run.

Use this for any field rename. **Do NOT do hard on-disk cutovers** тАФ rewrite-and-leave-no-fallback breaks shared-from-another-user files at the moment the migration lands.

## 6. Known traps / things I've stepped on

- **The AI proxy must NOT send `stream_options` on a non-streaming request.** OpenAI-compatible providers (Fireworks, our provider) HARD-REJECT `stream_options` with a 400 (`"stream_options is only valid when stream=true"`) whenever `stream` is not true. BeakerBot's agent loop runs `stream:false` to read `tool_calls` from one complete response, so a 2026-06-13 bug that included `stream_options` on non-streaming calls made EVERY BeakerBot turn 400 ("The model provider returned an error"). Fixed `f6fd2c311` (`frontend/src/app/api/ai/chat/route.ts`, only send it when `stream && billingOn`; non-streaming reads usage from the native completion JSON) with a regression guard in `route.test.ts`. This was INVISIBLE to gate checks (vitest mocks fetch, never hits a real provider) and only surfaced in a LIVE-MODEL Chrome test. LESSON: provider-contract bugs need a real-provider smoke test; the route deliberately does not echo the upstream body, so temporarily log `await upstream.clone().text()` in a disposable worktree to read the actual error. Two related facts from the same session: (1) the old `AI_VISION_MODEL` slug `llama-v3p2-11b-vision-instruct` is DEAD on Fireworks (404 Model not found) тАФ on our account only `kimi-k2p6` is multimodal (glm/deepseek reject images), and the prod Vercel var must be updated to a live model too (full path `accounts/fireworks/models/kimi-k2p6`, never shortened). (2) `NEXT_PUBLIC_DEV_AI_IN_DEMO` is HARD-GATED to `NODE_ENV !== "production"` (useAccountCapabilities.ts), so it does NOTHING on ANY Vercel deploy (Preview included тАФ Vercel builds are `NODE_ENV=production`); it only works in local `next dev`, so never set it in Vercel.

- **Parallel top-level sessions must each run in their own worktree, never share the main checkout.** The current git branch is a property of the working directory, not the session. So when several interactive sessions are all launched in `/Users/gnickles/Desktop/ResearchOS`, they share one `HEAD`, and a `git checkout` in any one of them silently drags every other session onto that branch. On 2026-06-08 one session ran `git checkout security/p1a-invite-key-out-of-email` to park inventory/sharing/mobile WIP, every other session in the main checkout was instantly on that branch too, and FleetView showed them all on `p1a-invite` instead of `main`. The parked work then had to be reconciled back. The rule going forward is to spawn each parallel top-level session in its own `git worktree` (use `scripts/new-session-worktree.sh <name>`), and keep the main checkout pinned to `main` as the integration tree. A running session's cwd is fixed at launch, so you cannot relocate one mid-flight; cycle existing sessions into worktrees as they wrap. **Reconciling parked work** (when it has already happened). Do it from a throwaway worktree off current `main`, cherry-pick only the genuinely-unmerged commits (find them with `git log --oneline --cherry-pick --right-only main...<branch>`, since `--cherry-pick` ignores commits already re-applied under a different SHA), verify, then land with `git merge --ff-only` at a moment when none of the touched files are dirty in the shared checkout. A whole-branch merge re-drags already-reconciled commits and conflicts on them; the cherry-pick-the-delta path stays clean.

- **The "drag image to trash deletes a sibling too" bug.** Fixed in commits `6d77e739` and `6f068e1d`. Root cause was `gcUnreferencedAttachments` not counting subfolder-style refs (`Images/{folder}/foo.png`) as protecting `foo.png` at the top of `Images/`. Fixed by extracting basenames at any depth. If you see similar "this code is dropping refs to legacy paths," look for `referencedRelativeNames` and `migrateNoteImages`.

- **`h-full` chain only works under a definite-height ancestor.** Non-fullscreen popups used `max-h-[90vh]` (no explicit height) and the editor's internal scroll broke. Fix: ensure `h-[Nvh]` or equivalent on the popup. (Commit `c0876e3d`.)

- **Cross-user dependency cascade is namespace-bounded тАФ but receivers now get a shift notification.** `shiftTask(taskId, тАж, owner)` still only walks deps in that one user's directory (cross-user writes would break the ownership invariant). What landed at `87b738d6` is the Option-C "notify-on-receive" half: every `tasksApi.move` that affects a task with `shared_with.length > 0` appends an entry to `users/<owner>/_shifted-alerts.json` (writer side), and every receiver scans her owners' sidecars at `NotificationBadge` mount via `sharingApi.scanShiftAlerts`, dedups against her local `_seen-shift-alerts.json`, and synthesizes a `ShiftAlertNotification` into the existing `_notifications.json` inbox. - **Field renames sometimes outpace callers.** When you see a typecheck error in `local-api.ts` for a Notes/Project/etc field that doesn't exist on the inferred type, it's usually because an interface in `lib/types.ts` was renamed by another agent and a caller wasn't updated. The typical fix touches **three layers in lockstep**: (1) the TypeScript interface in `frontend/src/lib/types.ts`, (2) the corresponding `*Api` adapter in `frontend/src/lib/local-api.ts` (read/write/normalize paths, including any lazy migration), and (3) the React component callers that pass the field. Grep the old name across all three before declaring the rename complete тАФ partial renames often typecheck because callers go through `any`-shaped intermediaries. **Note:** Zod runtime validation was consolidated out (commit `f1e3d7be`, 2026-05-13); types are now hand-written TS interfaces in `types.ts`. - **Drops over `<img>` need native capture-phase listeners тАФ React's `onDrop` doesn't win.** When a user drops a native OS file over a rendered `<img>` inside the markdown body, Chrome's per-element drop default ("replace image with file URL") fires BEFORE React's synthetic event delegation reaches inner element handlers. React's `onDrop` on the `<img>`, the markdown block, and the editor wrapper all silently miss the event; the file falls through to the window-level `GlobalDropGuard`. The fix that actually works: register a native `dragover`/`drop` listener via `useEffect` + `addEventListener(..., true)` (capture phase) on the editor's outer wrapper. - **Worktree edits silently land on the main checkout if you Read/Edit by main-checkout absolute path.** When operating in a worktree (`.claude/worktrees/<name>/`), `Read` and `Edit` use whatever absolute path you give them тАФ the harness does NOT redirect paths under `/Users/<u>/Desktop/ResearchOS/frontend/...` to the equivalent worktree path. So if you `Read` a file at the main-checkout path while exploring, every subsequent `Edit` to that file also goes to main. You'll find this out when `git status` inside the worktree returns clean but `git status` in main shows your changes. Recovery is easy (the diff is real, just on the wrong branch), but the cleaner habit is: in a worktree, either `cd` to the worktree first and use relative paths, or pass the explicit worktree-anchored absolute path. Hit this on 2026-05-13 mid-chip-sweep; nothing broke but a few seconds of confusion.

- **Next.js 16 client-component `params` are async.** A change from Next 15: in client components, route params are now a `Promise<{...}>` rather than a plain object. Code that does `const { slug } = params;` typechecks but throws at runtime. Discovered by the Demo v2 catch-all route bot on 2026-05-14; the implementer worked around it cleanly using `usePathname()` + string slice (no `params` access at all). For dynamic-segment client components, either `await` the Promise via `use(params)` from React, or sidestep `params` entirely with `usePathname()`. - **Verification bot's dev server running from a stale worktree.** When a verification bot probes `localhost:3001` (or wherever your dev server is), Next.js serves whatever the dev process's `cwd` was when started тАФ which can be a sub-bot's worktree that was checked out earlier in the day, not the current `main` checkout. The failure mode looks like "the feature didn't ship" (broken-looking pages, hung loading screens) rather than "the server is wrong" тАФ easy to misdiagnose. - **`/wiki/*` belongs to a dedicated wiki sub-bot, not the feature sub-bot.** The pattern is: feature ships in main, master summarizes what changed, master spawns a separate wiki-focused sub-bot to draft the page with the right voice plus screenshots. - **Server-side proxy rate limits are per-instance by default.** ResearchOS exposes 5 server-side proxy routes (`/api/calendar-feed`, `/api/telegram-file`, `/api/labarchives/fetch-image`, `/api/auth/labarchives/login`, `/api/auth/labarchives/refresh`). - **FSA `createWritable()` truncates the underlying file immediately, so a torn write leaves zero bytes, not "the old contents."** The default FSA write pattern is `createWritable()` then `write()` then `close()`. The truncate happens at `createWritable()` time, BEFORE any new bytes are committed. A tab close, polling-tab crash, or unhandled rejection between createWritable and close leaves the on-disk file as zero bytes. `readJson` returns null; the UI shows the file as missing; the user assumes a deletion bug. The fix lives in `fileService.writeJson` and `fileService.writeFileFromBlob` (both at [frontend/src/lib/file-system/file-service.ts](frontend/src/lib/file-system/file-service.ts)): write to `${path}.tmp` first, then `move()` to the final name (Chromium FSA atomic rename) so the failure mode becomes "old contents survive" instead of "file truncated to empty." If you add a new write path, route it through these methods or replicate the pattern. **Do not call `createWritable()` directly on the final file unless you have an explicit reason (streaming export, large append)**: the public `fileService.createWritable` helper is preserved for that case but bypasses the atomic guarantee. Non-Chromium browsers (Firefox, Safari) lack FSA `move()`; the helper falls back to `removeEntry(final)` then rewrite (non-atomic but the `.tmp` still acts as a checkpoint). Surfaced by the `_telegram.json` deletion investigation (2026-05-15); fixture and test coverage at [frontend/src/lib/file-system/__tests__/file-service-atomic.test.ts](frontend/src/lib/file-system/__tests__/file-service-atomic.test.ts).

- **Staged-but-uncommitted work in main checkout blocks ANY parallel-arc merge, even when the staged files don't overlap.** Surfaced by repeated PS UI ├Ч Onboarding v3 collisions during cross-arc merges (2026-05-20; three recurrences). Standard master workflow: cherry-pick source-only from a sub-bot branch via `git checkout <branch> -- <files>`, which STAGES the files in main checkout's index without committing. Between the staging step and the final commit, the index is dirty. If master then tries `git merge <other-branch>` during that window (to land a parallel arc's chip), git aborts with "Your local changes to the following files would be overwritten" тАФ even if the other branch and the staged files have ZERO file overlap. Git refuses to merge while there's a pending partial commit; it can't safely interleave a merge with the in-flight index state. **The aggregate behavior surprises an orchestrator thinking "my files don't touch theirs, safe to merge."** It's git's documented contract, but in a multi-arc workflow it's a structural footgun. **Resolution / discipline:** master-side cherry-pick + commit should be atomic. Don't pause between `git checkout <branch> -- <files>` and `git commit`. If a pause is unavoidable (e.g., need to investigate something mid-cherry-pick), either (a) commit a partial WIP and amend later, or (b) `git stash` the staged state so the index is clean. **Companion to 5950bcaf:** that entry covers sub-bots writing to main checkout instead of their worktree; this one covers master itself leaving staged work in main checkout between cherry-pick and commit. Both manifest as "dirty main checkout blocks parallel arcs" but root causes are distinct: 5950bcaf is sub-bot-CWD-confusion, this is master-workflow-pause. **Predictor:** any arc where master runs cherry-pick + commit as separate uninterruptible operations is safe; arcs where master pauses mid-pipeline (to investigate a sub-bot report, write a ┬з8 refresh, draft a follow-up brief, etc.) risk blocking other arcs' merges during the pause window.

- **Worktree spawn lacks `node_modules`; the symlink-for-verify shortcut needs explicit cleanup before commit.** Surfaced by repeated observation across the Onboarding v3 + Project Surface UI arcs (2026-05-20; three chip cycles). Sub-bots verifying their work via `npm run typecheck` / `lint` / `prebuild` / `vitest` from a freshly-spawned worktree hit "command not found" or `ENOENT` because Agent/`spawn_task` worktrees don't inherit `node_modules` from main checkout (it's gitignored + heavy). Three sub-bots independently converged on the same workaround: `ln -s ../../node_modules frontend/node_modules` (or the equivalent relative path), run verify, then `rm frontend/node_modules` BEFORE any git operation. **The cleanup step IS the trap.** If the symlink survives into `git add`, git stages it as a regular file (it follows the link by default), and the commit becomes a 50MB+ artifact tracking the entire `node_modules` tree. Always run `git status` after verify + before `git add`; a "new file: frontend/node_modules" or "modified: frontend/node_modules" line means cleanup was missed. **Cross-reference ┬з4 line 142:** the symlink works fine for `typecheck` / `lint` / `prebuild` / `vitest` but BREAKS under Turbopack / `npm run dev` ("Symlink node_modules is invalid, it points out of the filesystem root" тАФ Turbopack rejects out-of-root symlinks). Static verification workflows can use the symlink shortcut; live dev-server workflows need a real `npm install` in the worktree. **Master-side brief discipline:** when spawning a sub-bot whose verify pass needs npm scripts, the brief should explicitly cover (a) the symlink shortcut as a faster alternative to `npm install`, (b) the `rm` cleanup before commit, (c) the `git status` cross-check, and (d) the Turbopack incompatibility if any test path tries to boot the dev server.

- **Password-field paste-blocking is a browser-extension behavior, not a CSS concern.** Surfaced by the Telegram bot-token paste investigation (2026-05-15) + the followup at commit `5aed62dc` that superseded `bd5d52e8`'s broken type-toggle pattern. Chromium extensions (Claude in Chrome, password managers in their content-script form, some security extensions) attach `paste` listeners at `window` and intercept based on `focused field === type="password"`. The intercept overwrites the input value with watermark text like `"[BLOCKED: Sensitive key]"` тАФ confirmed via live DevTools event-bubble trace. **There is no CSP / Permissions-Policy / React-prop fix.** The block lives in the extension's isolated world; the page can't suppress it. **If you need both paste-compatibility AND visual masking on a non-credential field** (API keys, bot tokens, configuration secrets that aren't user-typed passwords), use `type="text"` with CSS `-webkit-text-security: disc` instead of `type="password"`. The CSS does the visual masking without triggering the extension heuristic. Trade-off: password-manager autofill won't fire for the field тАФ usually a feature, not a bug, since these aren't user credentials. Pattern landed at [TelegramPairingModal.tsx](frontend/src/components/TelegramPairingModal.tsx); mirror that shape for similar fields. **Anti-pattern**: a show/hide toggle that flips `type="password"` тЖФ `type="text"` makes the eye icon a paste-gate (users must click reveal first to paste), which contradicts every standard password-field UX. Use the CSS approach instead.

- **`main` parked in a parallel session's worktree blocks integration from the master checkout (2026-06-02).** With several sessions live, `main` is often checked out in another session's worktree (e.g. `seq-integ`), so `git switch main` in the master checkout fails with `fatal: 'main' is already used by worktree at ...`. If you then `git cherry-pick <sha>` while still on your own branch (which may already contain those commits), the pick silently resolves to "nothing to commit" and leaves a half-done cherry-pick state, recover with `git cherry-pick --abort`. You cannot advance `main` except where it is checked out. Options: (a) the worktree-owning session lands it, (b) ask Grant to free `main`, or (c) Grant lands it. Do NOT cherry-pick inside the other session's worktree on a hunch, a concurrent git-index operation there can corrupt that session. Prove the integration is clean before handing off: `git diff <fork-base> <main-HEAD> -- <changed-files>` empty means a conflict-free cherry-pick. To update a doc like this one against live main while main is parked, branch a fresh worktree off `main` (`git worktree add -b <name> <path> main`), edit there, and hand the branch off for landing.

- **A full-screen modal mounted inside a launcher's click-to-close backdrop must `stopPropagation` at its OWN root (2026-06-02, `3c14a772`).** The annotate editor (`ImageAnnotatorModal`) is `position: fixed` but rendered as a DOM descendant of `ImageMetadataPopup`'s backdrop `<div onClick={onClose}>`. `position: fixed` changes painting, NOT event bubbling, so every click inside (the rectangle tool, any tool, the canvas) bubbled to the backdrop and closed the whole popup (Grant: "I clicked the rectangle and it exited annotate mode"). Fix is one line on the modal root: `onClick={(e) => e.stopPropagation()}`, making the editor self-contained regardless of which launcher mounts it. Applies to any full-screen overlay launched from inside a backdrop-close container.

- **Browser-tab leader election must NOT steal the lock on focus (2026-06-02, `b8be3e27`).** The Telegram poller force-claimed the cross-tab localStorage lock on every window `focus` / `visibilitychange`, so with two ResearchOS tabs open, switching tabs ping-ponged the lock and the two `getUpdates` long-polls collided on Telegram's one-poll-per-bot rule (HTTP 409), dropping multi-step image flows mid-conversation. Fix: a focus only re-drains if THIS tab already holds (or can freely claim) the lock; it never steals from a live leader (`if (tryClaimLock(tabId)) pollController?.abort()`); failover still happens when the leader's lock goes stale. General rule for any cross-tab single-owner resource: stable leader + stale-failover, never steal-on-focus. The standby tab shows a calm "another tab" badge, not a "close your tabs" alarm, because results land in shared local data either way.

- **Correctness-critical numbers must be transcribed from a fetched authoritative source, never recalled (2026-06-02).** The nearest-neighbor primer-Tm (`lib/calculators/tm-nn.ts`) is a port of Biopython's `Bio.SeqUtils.MeltingTemp.Tm_NN`; the thermodynamic parameters (SantaLucia/Allawi NN table, von Ahsen salt-equivalent) were transcribed verbatim from the Biopython source curled to `/tmp` and read as RAW BYTES (not via the WebFetch summarizer, which can mangle numeric tables) and pinned by tests against Biopython's documented reference Tm values. Do this for any scientific/financial constant: fetch the real source, transcribe exactly, add a reference-value test. The OSS-credits generator follows the same rule for licenses, it reads the actual installed `package.json` `license` field, never guesses.

- **Adding an npm dependency is a bundle-weight plus shared-manifest decision (2026-06-02).** `mathjs` (added for the Scientific calculator) rode the synchronous `AppShell` shell chunk and added ~104 KB gzipped to EVERY page's first-load JS (not code-split). Swapped to `expr-eval-fork` (MIT, ~8 KB gz) for ~96 KB/page saved. Lessons: (1) profile bundle weight before adding a heavy lib to this local-first / no-backend app, a `next build` size delta is the proof; (2) isolate the engine behind a thin tested wrapper (`lib/calculators/scientific.ts`) so swapping it is a one-file change with the test suite as the contract; (3) `package.json` + `pnpm-lock.yaml` are shared manifests, commit a dep add atomically with the feature, and beware a concurrent session leaving the shared `node_modules` with a missing top-level hoist link (package intact in the `.pnpm` store), which breaks imports for everyone until `pnpm install`. Run binaries directly (`node_modules/.bin/tsc`, `node_modules/.bin/vitest`); `pnpm exec` triggers a dep-status reinstall that fails on the sharp/unrs-resolver build-scripts approval.

- **The vendored SeqViz files are `@ts-nocheck`, so `tsc` does NOT catch undefined-variable or out-of-scope references in them (2026-06-04).** `frontend/src/vendor/seqviz/Linear/{Primers,SeqBlock,Linear}.tsx` and `Circular/Circular.tsx` all open with `// @ts-nocheck` (vendored, kept out of strict typecheck). A clean `tsc --noEmit` is NOT a real check for these files. During the primer base-render redesign a reference to `seqFontSize` that was never destructured in `Linear.render` compiled clean and then crashed the ENTIRE viewer at runtime via SeqViz's error boundary ("seqFontSize is not defined"). After editing a vendored seqviz file, manually grep the enclosing render/function scope for every new identifier you reference and confirm it is actually defined or destructured before claiming done. Related styling trap in the same files, the shared `annotation` style (`vendor/seqviz/style.ts`) sets `fillOpacity: 0.7` AND `strokeWidth: 0.5`, and because an inline `style=` beats SVG presentation attributes, applying `style={annotation}` silently grays out any `fill` and overrides any `strokeWidth` set as an attribute. The primer base shapes deliberately drop that style and set a solid fill plus a real border width instead.

- **The base-level primer render spans four files that must stay in sync (2026-06-04).** SnapGene-style primer drawing lives in `lib/sequences/primer-base-layout.ts` (pure column-mapping math, strict + tested), `vendor/seqviz/Linear/Primers.tsx` (the SVG, annealing box + raised 5' tail box + pulled-back 3' arrowhead + bases + label), `vendor/seqviz/Linear/SeqBlock.tsx` (`primerRowHeight` sizes each primer row taller in the base view and adds a second box-lane for tailed tracks plus barb headroom; `primerRowsHaveTail` detects the tail), and `vendor/seqviz/Linear/Linear.tsx` (`blockHeight` must add the SAME per-track height so stacked blocks never clip). Change the geometry in one without the others and you get clipped boxes or off-screen overflow. The proportion constants (`baseColor`, `boxFill`, `borderW`, `headLen`, `barbRise`, `barbBack`) are named at the top of the primer element in `Primers.tsx` for live tuning. A primer with no stored oligo note (`/note "primer <SEQ>"`) gets its bases by pinning the layout directly to the recorded binding span (forward template, reverse-complemented for a reverse primer), NOT by re-searching, so a short or repeated region cannot make the bases land on a duplicate elsewhere. Full inheritor briefing in `docs/orchestrator-handoffs/2026-06-04-sequence-editor-handoff.md`.
