## 1. What ResearchOS is

A local-first research project management app for science labs:

- GANTT scheduling with dependency-aware date shifts.
- Lab notes + results: markdown editor with image strip, drag-drop attachments, image gallery picker.
- PCR protocol builder with reagent calculators.
- Methods library with reusable protocols.
- Multi-user shared folders (OneDrive / Dropbox / iCloud) with per-user accounts, password gates, and shared-task editing.
- Telegram bot ingestion → image inbox.
- External ICS calendar overlays.
- Lab domains + publication companion sites (built behind `LAB_SITES_ENABLED`, OFF/inert): a paid lab gets `researchos.app/<labslug>` and publishes citable companion-site pages (markdown + live ResearchOS figure/table/dataset blocks; datasets streamed from R2 via DuckDB; figures baked-on-publish for citation safety) AND bring-your-own static sites (zip upload or public-GitHub connect), served cookie-isolated at `<labslug>.research-os.com`. Hosted data is metered (pass-through) with a 30-day lapse-reclaim GC. See `[[project_lab_domains_companion_sites]]` + `docs/handoffs/2026-06-16-lab-domains-go-live-checklist.md`.

**Architecture:** Next.js 16 (App Router) + React 19 + TypeScript + Tailwind. **No backend.** All data lives in a folder on the user's disk, accessed via the **File System Access API**. The only server-side code is two thin proxy routes (`/api/telegram-file`, `/api/calendar-feed`) that exist purely to bypass CORS for those integrations.

**Hosted at:** [research-os.app](https://research-os.app/) (the live public domain, auto-deploys from `origin/main`). Also runs locally via `./start.sh` (frontend only; the old FastAPI backend was deleted).

## 3. The local-api surface

`frontend/src/lib/local-api.ts` is the most-touched file in the codebase (~2000+ lines). It exports a set of APIs that mirror what used to be the FastAPI backend:

- `projectsApi`, `tasksApi`, `dependenciesApi`, `methodsApi`, `pcrApi`, `goalsApi`, `eventsApi`, `purchasesApi`, `lab_linksApi`, `notesApi`, `inbox*`, `sharingApi`, `usersApi`, `labApi`, `attachmentsApi`, `filesApi` (was `githubApi`, kept aliased for legacy callers — renamed in commit ~`128033bf`).

Editable shared tasks: most mutating `tasksApi` methods (`update`, `move`, `delete`, `addMethod`, etc.) take an optional `owner` argument. When the popup is for a task that's `is_shared_with_me` + `shared_permission === "edit"`, the wrapper in `TaskDetailPopup` (`ownerScopedTasksApi(task)`) automatically threads `owner` through every call. `shiftTask` in `engine/shift.ts` is similarly owner-aware — when `owner` is set, every read and write (deps, tasks, project weekend lookups) is routed to that owner's directory.

## 4. Working conventions

### Product philosophy (the standing values, read these first)

These are the durable values every agent optimizes for. They override convenience and build-effort savings.

1. **Efficiency is the name of ResearchOS.** The whole product exists to save researchers time and effort. When two solutions are on the table, pick the one that is best for the USER's efficiency and experience, even when it is by far the most work for US to build. Build effort is never a reason to ship a worse experience. The amount of work it takes us is our problem to solve, not the user's. (Canonical example, 2026-06-14: for the reproduce-from-PDF figure-style match we chose the full figure picker with page thumbnails + drag-to-crop over the cheaper auto-send-all-pages shortcut, because a clean cropped figure gives the user a faithful match and zero wrong-figure failures. The shortcut was less work for us and a worse product, so it lost.)
2. **Never ship a number we have not validated.** (See the Data Hub "Principle 2, the most important one" rule in section 8. Every computed stat ships with its scipy/R/Prism validation case through the gate. This is the special-cased, most important instance of the broader rule, do not regress published or computed values.)

When a build decision trades user value against our effort, state the tradeoff plainly and default to user value, the way the figure-picker decision did.

`github.com/gnick18/ResearchOS` is a PUBLIC, clonable repo. ### Sub-agents (spawn vs inline)

Big chunks of work were done by spawning fresh agents in their own worktrees via the `Agent` tool (or its older alias `mcp__ccd_session__spawn_task`). The orchestration model is **flat**: the master bot dispatches sub-bots directly, processes their reports, cherry-picks commits, and manages the in-flight queue. There is no intermediate manager tier; sub-bots are single-purpose and do not coordinate with each other.

**Spawn when:**

- Multi-hour work that doesn't depend on this chat's state (audit sweeps, large refactors, isolated features).
- Anything where the user explicitly says "spawn an agent."

**Don't spawn — do it inline — when:**

- Single-file fixes.
- Anything that needs ongoing dialogue with the user.
- Trivial cleanup.

Reliable recipe for a spawn prompt:

- **Self-contained briefing.** Spawned agents don't see prior chat history. Include the project's one-liner, the specific files to touch, the current commit they should expect to see, and what's out-of-scope.
- **Scope guards.** Always say what NOT to do (don't touch on-disk JSON format, don't refactor unrelated things, don't migrate to a new component if scope is "add `title` attrs").
- **Reproduction recipe** if you can give one (`as Grant open Kritika's shared task — should …`).
- **Worktree bootstrap.** A fresh worktree has no `node_modules` — without it, `npx tsc` falls through to a placeholder ("This is not the tsc command you are looking for") and the verification gate is silently bypassed. **For tsc + vitest (no dev server): SYMLINK, never install** — `ln -s /Users/gnickles/Desktop/ResearchOS/frontend/node_modules frontend/node_modules` (and the root one). A bare `npm`/`pnpm install` once triggered a macOS TCC permission revocation that killed every running agent; the symlink is the rule for the verify gate. **For a LIVE DEV SERVER (Preview MCP / browser walkthrough), the symlink does NOT work** — Turbopack rejects a node_modules that escapes the worktree root ("Symlink node_modules is invalid, it points out of the filesystem root"), and an APFS COW clone (`cp -c -R`) of this repo's **pnpm** node_modules leaves a broken `node_modules/node_modules` self-symlink Turbopack also rejects (the route 500s on first compile). The only thing that works for a dev server is a real `pnpm install --prefer-offline` in the worktree, which is SAFE + fast (the TCC-kill incident did NOT recur across many isolated-worktree installs 2026-06-11; pnpm hardlinks from the global store). Use the committed helper **`scripts/worktree-dev.sh <worktree-frontend-dir> <port> [ENV=V ...]`** (detects a missing/broken node_modules, installs if needed, execs `next dev` with any flags like `NEXT_PUBLIC_DATAHUB_ENABLED=1`); point a `.claude/launch.json` entry's runtimeArgs at it, then `preview_start`. **pnpm is PINNED (2026-06-11):** `frontend/package.json` `"packageManager": "pnpm@10.34.3"`, so Corepack auto-switches any local pnpm to 10.34.3 (first run downloads it once). The dead `frontend/pnpm-workspace.yaml` was DELETED — do NOT re-add it; build-script approvals live in `package.json` `pnpm.onlyBuiltDependencies` (sharp, unrs-resolver) + `ignoredBuiltDependencies` (core-js). Before the pin, local pnpm 11 vs Vercel pnpm 10 was what kept showing that file as perpetually dirty.
- **Verification gate**: `cd frontend && npx tsc --noEmit` must pass with exit 0. Optionally `npx eslint src/`. Live test in `http://localhost:3000`.
- **Staleness check.** Tell them: "If you don't see `frontend/src/lib/local-api.ts` or recent commit `<hash>`, the worktree is stale — stop and report." This catches the case where the spawned worktree is branched off pre-FSA-migration state.

### Sub-bot anchor drift (2026-05-27, learned the hard way)

A spawned worktree branches from the commit `main` pointed at **when the agent was launched**, not live `main`. During a long session the orchestrator lands dozens of cherry-picks onto `main`, so by the time a sub-bot finishes, its anchor can be 30+ commits stale. This bit us repeatedly in one night. Two halves to the rule:

**Sub-bot side — rebase before committing.** Brief every spawned bot: "FIRST run `git merge main --no-edit` in your worktree root and confirm it succeeds before doing any work. If it conflicts non-trivially, STOP and report." A bot that rebases onto current `main` produces a commit that cherry-picks cleanly with no conflict resolution at all. The telegram-image-path bot did this and its cherry-pick was a clean fast-forward; the bots that didn't caused the messes below.

**Orchestrator side — never `-X theirs` a stale-anchor commit onto a fast-moving main.** `-X theirs` resolves every conflict by taking the cherry-picked (stale) side. When that side is older than `main`, it **silently reverts** whatever landed on `main` after the anchor. Concrete damage from one night:
- A telegram-tutorial cherry-pick reverted 424 lines of `batch-routing.test.ts` coverage that had landed post-anchor (tests still "passed" — on a shrunken set). Had to restore main's file and re-graft the new tests.
- A workbench-create cherry-pick auto-merged `step-machine.ts` from the stale base and produced **duplicate step ids** + a dropped `settings-tour-calendar` slot, which only surfaced as test failures. Had to revert the whole commit and re-apply the change surgically by hand on top of current main.

**Verification after every cherry-pick:** run `git show <hash> --stat` and sanity-check the line deltas. If a file you didn't expect to shrink lost hundreds of lines (especially a test file), that's anchor-drift clobbering, not an intended edit. Diff the file against `main`'s pre-cherry-pick version (`git show <main-before>:path | wc -l`) and recover the lost content. Better: prefer **merging the sub-bot's branch** over `cherry-pick -X theirs` when the branch was rebased onto main; reserve `-X theirs` for the case where you've already confirmed the only divergence is the intended change.

**Circular-import footgun (same night):** `step-registry → WikiPointerStep → TourBootstrap → step-registry` closed a cycle that Next.js tolerated but vitest's module loader did not — it silently broke the entire `step-bodies.test.tsx` import (0 tests ran), masking every step-body assertion for who-knows-how-long. When a registry imports a step that imports a bootstrap that imports the registry, extract the shared leaf helpers into a dependency-free module (the `wiki-pointer-nav-flag.ts` / `lib/query-client.ts` pattern). If a whole test file reports "no tests" instead of pass/fail, suspect an import cycle, not an empty file.

### Sub-bot worktree discipline (2026-05-29, current hard rules)

The non-negotiable preamble every worktree sub-bot brief must carry, learned across the cohort sessions:

- **`.test.tsx` (jsdom) tests are unreliable in a symlinked/COW worktree; trust `.test.ts` and re-check `.tsx` from main (2026-06-18).** The vitest config splits into a `node` project (`src/**/*.test.ts`, no jest-dom) and a `jsdom` project (`src/**/*.test.tsx`, jest-dom via `src/test-setup.ts`). In a worktree whose node_modules is symlinked or COW-cloned, the jsdom project's `@testing-library/jest-dom/vitest` extends a DIFFERENT `expect` instance than the tests use, so every `.test.tsx` fails with `Invalid Chai property: toBeInTheDocument` even though the files are untouched and pass on main. Do not chase this as a real regression. The `node`-project `.test.ts` files are unaffected and are the trustworthy gate in a worktree; verify component (`.tsx`) tests from the main checkout (or an isolated `pnpm install` worktree per the rule above).
- **NEVER `git stash`.** The stash is a single repo-wide resource shared across every worktree; one bot stashing has wiped another session in-progress stash. If a bot needs a clean tree it commits or discards its own exact paths, never stash.
- **Never pipe a gate/test/build command through `tail`** (or any pipe). The pipe exit code masks the real one, so a failing `tsc`/vitest reads as green. Use `cmd > /tmp/x.log 2>&1; echo "EXIT=$?"` then read the log. The login shell here runs with `set -e`, so also guard any command that can legitimately exit non-zero (a `grep` with no match) with `|| true`.
- **Avoid apostrophes in heredoc commit bodies.** Commits use `git commit -m "$(cat <<'EOF' ... EOF)"`; a stray apostrophe inside the quoted heredoc breaks the quoting and produces a garbled or failed commit. Reword to drop it, or use `git commit -F /tmp/msg.txt` for a body that must contain one.

These are the standard preamble; paste them into every worktree brief.

### Large static data assets → R2, never git, never fs-read `public/` on Vercel (2026-06-16)

- A multi-MB generated data asset (e.g. the ROR institution registry, ~28 MB) must NOT be committed to git (permanent history bloat) and must NOT be read at runtime via `fs.readFileSync(process.cwd()/public/...)`. On Vercel, `public/` is served by the CDN but is NOT in the serverless function filesystem, so the fs read silently fails and the feature is inert in prod (it "works" only in `next dev`, which has the full disk). - Pattern: host the asset (gzipped) on Cloudflare R2 — the same bucket + `R2_*` creds the relay already uses (`frontend/src/lib/sharing/relay/storage.ts`) — and fetch it server-side via an S3 client, gunzip, cache in module scope, degrade gracefully if unreachable. Do NOT cache a failed/empty fetch (so it self-heals once the asset is uploaded, no redeploy).
- `vercel env pull` does NOT return values for variables marked "Sensitive" (it writes the key with a blank value), so it cannot be used to obtain secrets like `R2_SECRET_ACCESS_KEY` for a local upload. Provide those creds another way (export in the shell) when running an upload script locally.

### Field migrations

When a field on Task / Method / Project / Note / etc. is renamed or restructured, follow the **lazy-normalize + on-demand-repair** pattern the cleanup pass landed (commit `147db270`, 2026-05-13). The whole point is that shared on-disk files from other users with legacy shapes keep working transparently — no flag-day cutovers, no broken receivers.

1. **Rename the field** in `frontend/src/lib/types.ts` and update every caller in lockstep. (See the §6 "Field renames" trap for the grep checklist — partial renames silently typecheck.)
2. **Add a `normalize<Entity>Record` helper** at the read boundary in `frontend/src/lib/local-api.ts`. Templates: `normalizeMethodRecord`, `normalizeTaskRecord`. The helper detects the legacy shape, rewrites it in-memory to the new shape, returns the normalized record. Apply on every read path (`.get`, `.list`, `fetchAllXIncludingShared`, …) so callers never see the legacy shape.
3. **Add a one-shot repair button** to `frontend/src/app/settings/page.tsx` under "Data maintenance." It iterates every stored file of that entity, normalizes, writes back. Report scanned / repaired / already-clean counts. Safe to re-run.

Use this for any field rename. **Do NOT do hard on-disk cutovers** — rewrite-and-leave-no-fallback breaks shared-from-another-user files at the moment the migration lands.

## 6. Known traps / things I've stepped on

- **The AI proxy must NOT send `stream_options` on a non-streaming request.** OpenAI-compatible providers (Fireworks, our provider) HARD-REJECT `stream_options` with a 400 (`"stream_options is only valid when stream=true"`) whenever `stream` is not true. BeakerBot's agent loop runs `stream:false` to read `tool_calls` from one complete response, so a 2026-06-13 bug that included `stream_options` on non-streaming calls made EVERY BeakerBot turn 400 ("The model provider returned an error"). Fixed `f6fd2c311` (`frontend/src/app/api/ai/chat/route.ts`, only send it when `stream && billingOn`; non-streaming reads usage from the native completion JSON) with a regression guard in `route.test.ts`. This was INVISIBLE to gate checks (vitest mocks fetch, never hits a real provider) and only surfaced in a LIVE-MODEL Chrome test. LESSON: provider-contract bugs need a real-provider smoke test; the route deliberately does not echo the upstream body, so temporarily log `await upstream.clone().text()` in a disposable worktree to read the actual error. Two related facts from the same session: (1) the old `AI_VISION_MODEL` slug `llama-v3p2-11b-vision-instruct` is DEAD on Fireworks (404 Model not found) — on our account only `kimi-k2p6` is multimodal (glm/deepseek reject images), and the prod Vercel var must be updated to a live model too (full path `accounts/fireworks/models/kimi-k2p6`, never shortened). (2) `NEXT_PUBLIC_DEV_AI_IN_DEMO` is HARD-GATED to `NODE_ENV !== "production"` (useAccountCapabilities.ts), so it does NOTHING on ANY Vercel deploy (Preview included — Vercel builds are `NODE_ENV=production`); it only works in local `next dev`, so never set it in Vercel.

- **Parallel top-level sessions must each run in their own worktree, never share the main checkout.** The current git branch is a property of the working directory, not the session. So when several interactive sessions are all launched in `/Users/gnickles/Desktop/ResearchOS`, they share one `HEAD`, and a `git checkout` in any one of them silently drags every other session onto that branch. On 2026-06-08 one session ran `git checkout security/p1a-invite-key-out-of-email` to park inventory/sharing/mobile WIP, every other session in the main checkout was instantly on that branch too, and FleetView showed them all on `p1a-invite` instead of `main`. The parked work then had to be reconciled back. The rule going forward is to spawn each parallel top-level session in its own `git worktree` (use `scripts/new-session-worktree.sh <name>`), and keep the main checkout pinned to `main` as the integration tree. A running session's cwd is fixed at launch, so you cannot relocate one mid-flight; cycle existing sessions into worktrees as they wrap. **Reconciling parked work** (when it has already happened). Do it from a throwaway worktree off current `main`, cherry-pick only the genuinely-unmerged commits (find them with `git log --oneline --cherry-pick --right-only main...<branch>`, since `--cherry-pick` ignores commits already re-applied under a different SHA), verify, then land with `git merge --ff-only` at a moment when none of the touched files are dirty in the shared checkout. A whole-branch merge re-drags already-reconciled commits and conflicts on them; the cherry-pick-the-delta path stays clean.

- **The "drag image to trash deletes a sibling too" bug.** Fixed in commits `6d77e739` and `6f068e1d`. Root cause was `gcUnreferencedAttachments` not counting subfolder-style refs (`Images/{folder}/foo.png`) as protecting `foo.png` at the top of `Images/`. Fixed by extracting basenames at any depth. If you see similar "this code is dropping refs to legacy paths," look for `referencedRelativeNames` and `migrateNoteImages`.

- **`h-full` chain only works under a definite-height ancestor.** Non-fullscreen popups used `max-h-[90vh]` (no explicit height) and the editor's internal scroll broke. Fix: ensure `h-[Nvh]` or equivalent on the popup. (Commit `c0876e3d`.)

- **OneDrive / iCloud picker can block JS for 30-60s.** `connect()` flips into a staged loading screen BEFORE calling `showDirectoryPicker` and yields two `requestAnimationFrame` ticks so React commits. CSS animations on the loading screen continue running during the JS block; `setInterval`-driven counters don't. See `StagedLoadingScreen.tsx`.

- **FSA handles persist via IndexedDB but permission grants don't.** On reload, call `handle.queryPermission({ mode: "readwrite" })` first; if `"granted"`, reconnect silently. Otherwise show a "Continue" button that fires `requestPermission` (needs user gesture). See `reconnectWithStoredHandle` in `file-system-context.tsx`.

- **Per-user ID collisions.** Grant's task id 1 and Kritika's task id 1 are different tasks. Always use `taskKey(task)` for in-memory uniqueness.

- **Cross-user dependency cascade is namespace-bounded — but receivers now get a shift notification.** `shiftTask(taskId, …, owner)` still only walks deps in that one user's directory (cross-user writes would break the ownership invariant). What landed at `87b738d6` is the Option-C "notify-on-receive" half: every `tasksApi.move` that affects a task with `shared_with.length > 0` appends an entry to `users/<owner>/_shifted-alerts.json` (writer side), and every receiver scans her owners' sidecars at `NotificationBadge` mount via `sharingApi.scanShiftAlerts`, dedups against her local `_seen-shift-alerts.json`, and synthesizes a `ShiftAlertNotification` into the existing `_notifications.json` inbox. The notification offers "View task" / "Ignore" — it does NOT auto-realign a dependent task because dep edges are intra-namespace today (`dep.parent_id` / `child_id` carry no owner; there is no cross-user dep to walk). If you're adding a new code path that calls `shiftTask` or writes to a task, make sure it goes through `tasksApi.move` so the sidecar gets appended — direct `tasksStore.save` bypasses the alert. **Where the alert lives** is the namespace `shiftTask` ran in (= the `owner` parameter, fallback current user); `shifted_by_user` is the actor (= current user), which is how a receiver filters "morgan-shifted-alex's-task vs alex-shifted-her-own" without double-noise. See `frontend/src/lib/shift-alerts.test.ts` for the dedup invariants the receiver scan relies on.

- **Field renames sometimes outpace callers.** When you see a typecheck error in `local-api.ts` for a Notes/Project/etc field that doesn't exist on the inferred type, it's usually because an interface in `lib/types.ts` was renamed by another agent and a caller wasn't updated. The typical fix touches **three layers in lockstep**: (1) the TypeScript interface in `frontend/src/lib/types.ts`, (2) the corresponding `*Api` adapter in `frontend/src/lib/local-api.ts` (read/write/normalize paths, including any lazy migration), and (3) the React component callers that pass the field. Grep the old name across all three before declaring the rename complete — partial renames often typecheck because callers go through `any`-shaped intermediaries. **Note:** Zod runtime validation was consolidated out (commit `f1e3d7be`, 2026-05-13); types are now hand-written TS interfaces in `types.ts`. If runtime validation is ever wanted again, reintroduce it surgically at the call site rather than as a parallel schema file.

- **Drops over `<img>` need native capture-phase listeners — React's `onDrop` doesn't win.** When a user drops a native OS file over a rendered `<img>` inside the markdown body, Chrome's per-element drop default ("replace image with file URL") fires BEFORE React's synthetic event delegation reaches inner element handlers. React's `onDrop` on the `<img>`, the markdown block, and the editor wrapper all silently miss the event; the file falls through to the window-level `GlobalDropGuard`. The fix that actually works: register a native `dragover`/`drop` listener via `useEffect` + `addEventListener(..., true)` (capture phase) on the editor's outer wrapper. Capture fires top-down BEFORE inner elements get the event. Pattern is in `frontend/src/components/LiveMarkdownEditor.tsx` around line 575 (added by `4ae53082`, 2026-05-13). Don't waste time trying React-side fixes for drops on `<img>` — skip straight to native capture-phase.

- **Worktree edits silently land on the main checkout if you Read/Edit by main-checkout absolute path.** When operating in a worktree (`.claude/worktrees/<name>/`), `Read` and `Edit` use whatever absolute path you give them — the harness does NOT redirect paths under `/Users/<u>/Desktop/ResearchOS/frontend/...` to the equivalent worktree path. So if you `Read` a file at the main-checkout path while exploring, every subsequent `Edit` to that file also goes to main. You'll find this out when `git status` inside the worktree returns clean but `git status` in main shows your changes. Recovery is easy (the diff is real, just on the wrong branch), but the cleaner habit is: in a worktree, either `cd` to the worktree first and use relative paths, or pass the explicit worktree-anchored absolute path. Hit this on 2026-05-13 mid-chip-sweep; nothing broke but a few seconds of confusion.

- **Next.js 16 client-component `params` are async.** A change from Next 15: in client components, route params are now a `Promise<{...}>` rather than a plain object. Code that does `const { slug } = params;` typechecks but throws at runtime. Discovered by the Demo v2 catch-all route bot on 2026-05-14; the implementer worked around it cleanly using `usePathname()` + string slice (no `params` access at all). For dynamic-segment client components, either `await` the Promise via `use(params)` from React, or sidestep `params` entirely with `usePathname()`. Server components are unaffected — they can `await params` normally.

- **Verification bot's dev server running from a stale worktree.** When a verification bot probes `localhost:3001` (or wherever your dev server is), Next.js serves whatever the dev process's `cwd` was when started — which can be a sub-bot's worktree that was checked out earlier in the day, not the current `main` checkout. Symptom on 2026-05-14: verification bot reported ALL 7 Demo v2 recipes INCONCLUSIVE because the bundle was missing `FloatingLeaveDemoButton.tsx`, `OpenDocsButton.tsx`, `TryInDemo.tsx`, and the catch-all `[[...slug]]/page.tsx` route — those existed in `main` (`74ecd115`, `80471925`) but not in the worktree the dev server happened to be in (`competent-tesla-1c0608` at `412a9fe0`). The failure mode looks like "the feature didn't ship" (broken-looking pages, hung loading screens) rather than "the server is wrong" — easy to misdiagnose. Diagnose by `ps -o pid,command,cwd <pid>` on the dev process or by grepping for one of the merge SHAs in the served bundle. Fix is just `kill <pid>` + restart from the right cwd. Bot brief should ideally tell the bot to verify the bundle freshness first (e.g. check that a known-new component name appears in the page source) and INCONCLUSIVE-out the whole run cleanly if not.

- **`/wiki/*` belongs to a dedicated wiki sub-bot, not the feature sub-bot.** The pattern is: feature ships in main, master summarizes what changed, master spawns a separate wiki-focused sub-bot to draft the page with the right voice plus screenshots. - **FSA `createWritable()` truncates the underlying file immediately, so a torn write leaves zero bytes, not "the old contents."** The default FSA write pattern is `createWritable()` then `write()` then `close()`. The truncate happens at `createWritable()` time, BEFORE any new bytes are committed. A tab close, polling-tab crash, or unhandled rejection between createWritable and close leaves the on-disk file as zero bytes. `readJson` returns null; the UI shows the file as missing; the user assumes a deletion bug. The fix lives in `fileService.writeJson` and `fileService.writeFileFromBlob` (both at [frontend/src/lib/file-system/file-service.ts](frontend/src/lib/file-system/file-service.ts)): write to `${path}.tmp` first, then `move()` to the final name (Chromium FSA atomic rename) so the failure mode becomes "old contents survive" instead of "file truncated to empty." Cross-check by running `git -C <worktree-abs-path> status` AND `git -C /Users/<u>/Desktop/ResearchOS status` from the bot's session; if the worktree shows clean but main shows the bot's changes, the bot has been writing to the wrong tree the whole time. **Resolution if detected mid-run:** abort the chip and report immediately. Do NOT attempt to copy-recover files between paths, that compounds the confusion and produces stale state in two places. Master will either (a) cherry-pick from wherever the sub-bot ends up landing committed work, or (b) treat the in-main-checkout state as never-having-happened and re-dispatch. **Mitigation in every future chip brief** (codified after this arc): include the verification clause `"VERIFY YOUR WORKTREE PATH BEFORE ANY FILE EDIT. Run pwd first; ensure it matches your assigned worktree (not /Users/gnickles/Desktop/ResearchOS). Use git -C <worktree-abs-path> exclusively for git commands, never cd + bare git. If you find files in main checkout's path during your run, ABORT and report immediately, do not attempt copy-recover."` The trap recurred three times this arc precisely because the chip briefs implied "use worktree" but didn't make verification a hard gate. **Outcomes from the three recurrences this arc:** Follow-up A's sub-bot self-detected via `git -C` cross-check after a few edits, copied files into its worktree, reverted main, re-verified, committed clean to `claude/pedantic-hellman-acb59d` @ `b8ac4ee1`. Follow-up B's sub-bot hit the same trap and recovered identically, landing on `claude/happy-heyrovsky-9561a7` @ `931ae36c`. P2b's sub-bot hit the trap and did NOT self-recover; its WIP (W1-W9 step files + import additions to `OnboardingWizardV3.tsx`) ended up stranded in main checkout uncommitted while two parallel sub-bots reported around it, requiring master to stash the WIP, complete unrelated cherry-picks, and unstash to restore the state for the eventual proper-branch landing. **Trap variant relationship:** this is the shell-level sibling of the existing "Worktree edits silently land on the main checkout if you Read/Edit by main-checkout absolute path" trap (above). That earlier entry covered the Read/Edit-tool variant within a single bot session; this entry covers the parallel-bot / shell-CWD-bounce variant where the bot has its own session but the shell sub-process flips trees on `cd`. Same root cause (absolute paths into main checkout silently work even from a worktree-rooted session), different surface (shell vs Read/Edit tool). **Dispatcher-side complement (codified 2026-05-20 after a wiki copy-fix chip recurrence):** the bot's `pwd` verification clause does NOT help if the brief itself cites file paths anchored to main checkout. A bot dispatched to a worktree that reads "fix the callout at `/Users/gnickles/Desktop/ResearchOS/frontend/src/app/wiki/...`" will Read/Edit that exact main-checkout path regardless of how diligently it pwd-checked. **Brief-writing discipline (master side):** when citing file paths in a chip brief for a worktree-bound sub-bot, use repo-relative paths (`frontend/src/app/wiki/features/experiments/page.tsx`) OR the sub-bot's specific worktree absolute path (`/Users/gnickles/Desktop/ResearchOS/.claude/worktrees/<name>/frontend/...`). NEVER the main-checkout absolute path. Repo-relative is preferred because it survives copy-paste between worktrees and reads naturally as "from your worktree root." Recurrence at `55aeff85` (2026-05-20): a chip brief to a wiki copy-fix sub-bot in worktree `tender-lovelace-7481b4` cited main-checkout absolute paths; the sub-bot pwd-verified worktree correctly, then misrouted the edit anyway because Read/Edit followed the brief's literal path strings. Single-file fix and recoverable (revert main, re-apply in worktree, single commit) but the trap is real and structural to how briefs reference files.

- **Parallel-arc AI Helper autogen collides at merge time. Resolve by delete-both + regenerate, not pick-a-side.** Surfaced by the Project Surface UI × Onboarding v3 simultaneous-arc-merge collision (2026-05-20) on `frontend/public/ai-helper/{full,lean,minimal}.md` + `manifest.json`. The AI Helper prebuild (`npm run --prefix frontend prebuild`) scans the WHOLE codebase to generate the four autogen files describing every API surface, type, schema. The output reflects whatever source code exists when prebuild runs. When two parallel arcs each touch AI-Helper-scanned surfaces (anything affecting `projectsApi` / `tasksApi` / `sidecar.ts` / `types.ts` / new exports under `lib/`), each arc independently runs prebuild on its branch and commits an autogen file that reflects only ITS view of the codebase. At merge time both files diverge — neither is "wrong" but neither is right after merge either. Git can't pick a winner. **Resolution:** `git checkout --theirs frontend/public/ai-helper/*` (the `--theirs` vs `--ours` choice is irrelevant — both get nuked next), then `npm run --prefix frontend prebuild` to regenerate canonical autogen reflecting BOTH arcs' source state. Confirm `git diff --stat HEAD` shows only the 4 ai-helper/* files post-prebuild. Then stage + commit the merge. Picking one arc's version + relying on the next prebuild to overwrite is the anti-pattern: it ships stale autogen until the next AI Helper-touching commit (could be days), during which time the AI Helper prompt advertises an inconsistent API surface to the user's Claude/ChatGPT/Gemini. **Collision predictor (refined after 4 data points 2026-05-20):** parallel arcs only collide here when BOTH have touched **TYPE contributions to the AI-Helper-scanned surface** — not just "any new lib/ export." The discriminator is whether the change adds a new TYPE that the AI Helper's surface-scanner picks up (sidecar field, new API method signature with new payload shape, new exported interface in a public API module, new entry in `types.ts`) versus pure utility/UI code the scanner skips. **Data points behind the refinement:** PS UI P2 (`projectsApi.getOverview/setOverview` — new API methods with new payload shapes) + v3 P2a (`feature-picks-tabs.ts` + sidecar.ts schema extension) → both BUMPED schema_hash, COLLIDED. PS UI P5 (`lib/project-activity/event-log.ts` — new module but only utility-function exports, no types in the AI-Helper-scanned surface) → schema_hash UNCHANGED, no collision. PS UI P6/P7/P8 (all UI-only edits) → schema_hash UNCHANGED, no collisions. Future master orchestrating multiple parallel arcs: anticipate this trap when phases touch `lib/types.ts` / `lib/local-api.ts` method signatures / `lib/onboarding/sidecar.ts` / new per-API modules under `lib/` that EXPORT TYPES. Phases that touch only UI components, internal helpers, or utility-function lib modules without type exports are collision-safe.

- **Source-only cherry-pick via `git checkout <branch> -- <file>` is WHOLE-FILE REPLACE, not diff-apply — reverts main's intervening edits when both bot and main modified the same file.** Surfaced by Onboarding v4 P0 cherry-pick 2026-05-21: bot's branch rooted at `f21800f3` adding `shareTaskAs` to `local-api.ts`; main at `c0c0ffa1` (-8 commits) had also modified `local-api.ts` with Project Surface P2's `getOverview`/`setOverview` + P5's `recordProjectActivity` emissions. Master ran `git checkout worktree-agent-X -- frontend/src/lib/local-api.ts` to source-only cherry-pick the bot's additions. The whole-file replace pulled in bot's STALE version of `local-api.ts`, OVERWRITING Project Surface's intervening work (-118 lines of `getOverview`/`setOverview`/`recordProjectActivity` wiring). Caught via diff stat `+197/-118` on a commit that was supposed to be `+97 net` per bot's report; reverted at `8f4ba897` + redone via `git cherry-pick 01aedf85` (diff-apply with 3-way merge) which correctly preserved both sides' changes. **The lesson:** `git checkout <branch> -- <file>` works ONLY when (a) the file is entirely new on the bot's branch, OR (b) main hasn't modified that file during the stale interval. When both bot AND main modified the same file, use `git cherry-pick <SHA>` instead — it applies the bot's DIFF rather than replacing main's file wholesale, and resolves the 3-way merge correctly. **For autogen files that both sides touched**, the cherry-pick will conflict on them; resolve via the b6c6dc73 pattern (`git checkout --theirs ai-helper/*` then `npm run prebuild` to regenerate from the merged source tree). **Predictor:** any time a stale-branch-rooted sub-agent's report shows the target file modified by `+N -M` (not just `+N` net), main likely has intervening changes to that file. Run `git diff <merge-base> <main> -- <target-file>` before cherry-picking to confirm; use `git cherry-pick <SHA>` if main shows any changes. The whole-file-replace pattern from 45c4bb88's source-only cherry-pick discipline is still correct for new files + untouched-in-main files; this entry refines the discipline for the both-sides-modified case the original entry didn't anticipate.

- **Staged-but-uncommitted work in main checkout blocks ANY parallel-arc merge, even when the staged files don't overlap.** Surfaced by repeated PS UI × Onboarding v3 collisions during cross-arc merges (2026-05-20; three recurrences). Standard master workflow: cherry-pick source-only from a sub-bot branch via `git checkout <branch> -- <files>`, which STAGES the files in main checkout's index without committing. Between the staging step and the final commit, the index is dirty. If master then tries `git merge <other-branch>` during that window (to land a parallel arc's chip), git aborts with "Your local changes to the following files would be overwritten" — even if the other branch and the staged files have ZERO file overlap. Git refuses to merge while there's a pending partial commit; it can't safely interleave a merge with the in-flight index state. **The aggregate behavior surprises an orchestrator thinking "my files don't touch theirs, safe to merge."** It's git's documented contract, but in a multi-arc workflow it's a structural footgun. **Resolution / discipline:** master-side cherry-pick + commit should be atomic. Don't pause between `git checkout <branch> -- <files>` and `git commit`. If a pause is unavoidable (e.g., need to investigate something mid-cherry-pick), either (a) commit a partial WIP and amend later, or (b) `git stash` the staged state so the index is clean. **Companion to 5950bcaf:** that entry covers sub-bots writing to main checkout instead of their worktree; this one covers master itself leaving staged work in main checkout between cherry-pick and commit. Both manifest as "dirty main checkout blocks parallel arcs" but root causes are distinct: 5950bcaf is sub-bot-CWD-confusion, this is master-workflow-pause. **Predictor:** any arc where master runs cherry-pick + commit as separate uninterruptible operations is safe; arcs where master pauses mid-pipeline (to investigate a sub-bot report, write a §8 refresh, draft a follow-up brief, etc.) risk blocking other arcs' merges during the pause window.

- **Worktree spawn lacks `node_modules`; the symlink-for-verify shortcut needs explicit cleanup before commit.** Surfaced by repeated observation across the Onboarding v3 + Project Surface UI arcs (2026-05-20; three chip cycles). Sub-bots verifying their work via `npm run typecheck` / `lint` / `prebuild` / `vitest` from a freshly-spawned worktree hit "command not found" or `ENOENT` because Agent/`spawn_task` worktrees don't inherit `node_modules` from main checkout (it's gitignored + heavy). Three sub-bots independently converged on the same workaround: `ln -s ../../node_modules frontend/node_modules` (or the equivalent relative path), run verify, then `rm frontend/node_modules` BEFORE any git operation. **The cleanup step IS the trap.** If the symlink survives into `git add`, git stages it as a regular file (it follows the link by default), and the commit becomes a 50MB+ artifact tracking the entire `node_modules` tree. Always run `git status` after verify + before `git add`; a "new file: frontend/node_modules" or "modified: frontend/node_modules" line means cleanup was missed. **Cross-reference §4 line 142:** the symlink works fine for `typecheck` / `lint` / `prebuild` / `vitest` but BREAKS under Turbopack / `npm run dev` ("Symlink node_modules is invalid, it points out of the filesystem root" — Turbopack rejects out-of-root symlinks). Static verification workflows can use the symlink shortcut; live dev-server workflows need a real `npm install` in the worktree. **Master-side brief discipline:** when spawning a sub-bot whose verify pass needs npm scripts, the brief should explicitly cover (a) the symlink shortcut as a faster alternative to `npm install`, (b) the `rm` cleanup before commit, (c) the `git status` cross-check, and (d) the Turbopack incompatibility if any test path tries to boot the dev server.

- **Password-field paste-blocking is a browser-extension behavior, not a CSS concern.** Surfaced by the Telegram bot-token paste investigation (2026-05-15) + the followup at commit `5aed62dc` that superseded `bd5d52e8`'s broken type-toggle pattern. Chromium extensions (Claude in Chrome, password managers in their content-script form, some security extensions) attach `paste` listeners at `window` and intercept based on `focused field === type="password"`. The intercept overwrites the input value with watermark text like `"[BLOCKED: Sensitive key]"` — confirmed via live DevTools event-bubble trace. **There is no CSP / Permissions-Policy / React-prop fix.** The block lives in the extension's isolated world; the page can't suppress it. **If you need both paste-compatibility AND visual masking on a non-credential field** (API keys, bot tokens, configuration secrets that aren't user-typed passwords), use `type="text"` with CSS `-webkit-text-security: disc` instead of `type="password"`. The CSS does the visual masking without triggering the extension heuristic. Trade-off: password-manager autofill won't fire for the field — usually a feature, not a bug, since these aren't user credentials. Pattern landed at [TelegramPairingModal.tsx](frontend/src/components/TelegramPairingModal.tsx); mirror that shape for similar fields. **Anti-pattern**: a show/hide toggle that flips `type="password"` ↔ `type="text"` makes the eye icon a paste-gate (users must click reveal first to paste), which contradicts every standard password-field UX. Use the CSS approach instead.

- **`main` parked in a parallel session's worktree blocks integration from the master checkout (2026-06-02).** With several sessions live, `main` is often checked out in another session's worktree (e.g. `seq-integ`), so `git switch main` in the master checkout fails with `fatal: 'main' is already used by worktree at ...`. If you then `git cherry-pick <sha>` while still on your own branch (which may already contain those commits), the pick silently resolves to "nothing to commit" and leaves a half-done cherry-pick state, recover with `git cherry-pick --abort`. You cannot advance `main` except where it is checked out. Options: (a) the worktree-owning session lands it, (b) ask Grant to free `main`, or (c) Grant lands it. Do NOT cherry-pick inside the other session's worktree on a hunch, a concurrent git-index operation there can corrupt that session. Prove the integration is clean before handing off: `git diff <fork-base> <main-HEAD> -- <changed-files>` empty means a conflict-free cherry-pick. - **A full-screen modal mounted inside a launcher's click-to-close backdrop must `stopPropagation` at its OWN root (2026-06-02, `3c14a772`).** The annotate editor (`ImageAnnotatorModal`) is `position: fixed` but rendered as a DOM descendant of `ImageMetadataPopup`'s backdrop `<div onClick={onClose}>`. `position: fixed` changes painting, NOT event bubbling, so every click inside (the rectangle tool, any tool, the canvas) bubbled to the backdrop and closed the whole popup (Grant: "I clicked the rectangle and it exited annotate mode"). Fix is one line on the modal root: `onClick={(e) => e.stopPropagation()}`, making the editor self-contained regardless of which launcher mounts it. Applies to any full-screen overlay launched from inside a backdrop-close container.

- **Browser-tab leader election must NOT steal the lock on focus (2026-06-02, `b8be3e27`).** The Telegram poller force-claimed the cross-tab localStorage lock on every window `focus` / `visibilitychange`, so with two ResearchOS tabs open, switching tabs ping-ponged the lock and the two `getUpdates` long-polls collided on Telegram's one-poll-per-bot rule (HTTP 409), dropping multi-step image flows mid-conversation. General rule for any cross-tab single-owner resource: stable leader + stale-failover, never steal-on-focus. The standby tab shows a calm "another tab" badge, not a "close your tabs" alarm, because results land in shared local data either way.

- **Correctness-critical numbers must be transcribed from a fetched authoritative source, never recalled (2026-06-02).** The nearest-neighbor primer-Tm (`lib/calculators/tm-nn.ts`) is a port of Biopython's `Bio.SeqUtils.MeltingTemp.Tm_NN`; the thermodynamic parameters (SantaLucia/Allawi NN table, von Ahsen salt-equivalent) were transcribed verbatim from the Biopython source curled to `/tmp` and read as RAW BYTES (not via the WebFetch summarizer, which can mangle numeric tables) and pinned by tests against Biopython's documented reference Tm values. Do this for any scientific/financial constant: fetch the real source, transcribe exactly, add a reference-value test. The OSS-credits generator follows the same rule for licenses, it reads the actual installed `package.json` `license` field, never guesses.

- **Adding an npm dependency is a bundle-weight plus shared-manifest decision (2026-06-02).** `mathjs` (added for the Scientific calculator) rode the synchronous `AppShell` shell chunk and added ~104 KB gzipped to EVERY page's first-load JS (not code-split).
