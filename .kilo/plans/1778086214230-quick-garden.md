# Fix: Overdue Tasks Not Populating After Folder Link

## Root Cause

**Task JSON files in storage do NOT have `end_date` stored.** They only have `start_date` and `duration_days`. 

When the backend was used, `end_date` was computed dynamically by the Python API (see `backend/app/routers/tasks.py` line 86). But in the new frontend-only mode, `JsonStore` reads raw JSON files without computing `end_date`.

### Evidence

From task file `37.json`:
```json
{
  "project_id": 4,
  "name": "Figure out where A. nidulans and S. livens are",
  "start_date": "2026-03-04",
  "duration_days": 1,
  "is_complete": false,
  // NO end_date field!
}
```

The overdue filter fails because:
```javascript
const overdue = activeTasks.filter(
  (t) => t.end_date < today && !t.is_complete
  // t.end_date is undefined, so undefined < "2026-05-06" is false
);
```

## Solution

Compute `end_date` when reading tasks in `local-api.ts`. Two approaches:

### Option A: Compute `end_date` in `fetchAllTasks` (Recommended)
Add a transformation step that computes `end_date` from `start_date` + `duration_days` for all tasks.

### Option B: Compute `end_date` in `JsonStore.query` 
More invasive but would apply to all entity types.

## Implementation Steps (Option A)

1. In `frontend/src/lib/local-api.ts`, modify `fetchAllTasks`:

**Before:**
```javascript
export const fetchAllTasks = async () => {
  return tasksStore.listAll();
};
```

**After:**
```javascript
export const fetchAllTasks = async () => {
  const tasks = await tasksStore.listAll();
  // Compute end_date for tasks that don't have it stored
  return tasks.map(t => ({
    ...t,
    end_date: t.end_date || formatDate(computeEndDate(parseDate(t.start_date), t.duration_days, false))
  }));
};
```

2. Also update `fetchAllTasksIncludingShared` similarly.

3. Consider updating `tasksApi.listByProject` as well to ensure consistency.

## Files to Modify

1. **`frontend/src/lib/local-api.ts`**
   - Import `computeEndDate`, `parseDate`, `formatDate` (already imported at line 6)
   - Modify `fetchAllTasks` to compute `end_date`
   - Modify `fetchAllTasksIncludingShared` to compute `end_date`
   - Modify `tasksApi.listByProject` to compute `end_date`

## Verification

## Long-term Fix (Optional)

Consider a one-time migration script that adds `end_date` to all existing task JSON files, so we don't need to compute it on every read.
