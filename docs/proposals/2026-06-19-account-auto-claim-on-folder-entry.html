Auto-claim your account on folder entry proposal

Why this exists

You signed in, opened a fresh folder, and the Share button never appeared. The cause is two correct-on-their-own pieces colliding.

  • Deferred mint (by design). Under the multi-folder identity model, a signed-in user entering a fresh folder enters with no keypair on purpose. The code comment (UserLoginScreen.tsx step 8) says the keypair is "minted on demand at the first sharing action... which surfaces the recovery code then." The intent was good, never mint a recovery-locked identity the user never saw the code for.
  • Share is gated on identity-ready. Every Share button checks canShare, which is true only once the identity is ready. With the mint deferred, canShare is false, so the Share button is hidden.
The dead zone. The "first sharing action" that was supposed to mint the key can never be reached, because the affordance that would trigger it is hidden until the key exists. The only escape is a buried Settings card ("Finish setting up your account"). A signed-in user is silently unable to share and is not told why on the surface where they tried.

Your direction: claiming should happen automatically when a signed-in user enters a folder, and if the folder was claimed by a different account, warn and let the user decide whether to take ownership.

Proposed flow: signed-in user enters a folder

One branch per folder-claim state. "Claimed" is read from the identity sidecar (fingerprint + optional bound email) and compared against the signed-in account's directory record.

S1 · Unclaimed
no identity sidecar
new Auto-establish the identity on entry (mint keypair, bind to the signed-in account, publish the findable profile), then show a one-time recovery-code card. No manual button. canShare true immediately.
Today: enters with no key, Share hidden, setup only via Settings.
S2 · Mine, key on this device
unchanged Reuse the on-device keypair silently (a reference sidecar). Already works. canShare true.
S3 · Mine, key not here
published from another device
exists, make automatic Auto-open the cross-device restore gate (enter recovery code), unwrap the canonical keypair, write a reference sidecar. The restore path exists today; ensure it fires automatically on entry rather than dead-ending.
S4 · Claimed by a different account
new Warn clearly: "This folder was set up by @other. You are signed in as @you." Offer Take ownership (re-key the folder to your account), Use a different folder, and Continue without sharing. Never silently re-key.
Today: a non-matching local key silently falls through to minting a fresh per-folder key, no warning, no choice.

Shared-file handling on takeover (S4)

Grant's requirement. When a different OAuth account takes over a folder, the new account does not hold the keys or permission for files that were shared to the prior identity. Those shares were encrypted and addressed to the old account, so the new account genuinely cannot decrypt or view them. Retaining another account's shared documents is both a privacy and a correctness problem, so takeover must detect, count, warn, and on confirm remove the local copies. The user's own (non-shared) files are never touched.

How shared files land today (audit)

There are two distinct delivery paths, and only one is recipient-encrypted. The takeover removal targets only the recipient-encrypted one.

Cross-boundary share
relay + R2, sealed
A bundle is sealed to the recipient's X25519 public key with sealToRecipient() and opened with openSealed() (frontend/src/lib/sharing/encryption.ts), fetched via receiveRawShare() (frontend/src/lib/sharing/relay/client.ts), then imported into the recipient's own folders as records under users/<currentUser>/{notes,projects,methods,experiments,tasks,sequences}/<id>.json. The import stamps three provenance fields, received_from, received_from_fingerprint, and received_at (note-transfer.ts:376, sequence-transfer.ts:246, import/apply.ts:56; typed in types.ts:697/1142/3166/3855/3917). This is the recipient-permissioned set. It was sealed to the prior identity's keypair; the new account cannot decrypt a re-fetch, and is not the intended recipient.
Intra-lab share
same folder, by reference
users/<u>/_shared_with_me.json (SharedManifest: version + projects/tasks/methods entry arrays; read/write at local-api.ts:8111/8122) points at records that physically live in another user's directory inside the same folder. These are not sealed to a keypair; they are plain cross-user references within one lab folder. Not in scope for removal: they belong to the folder's other members, not to the prior owning account, and the takeover is about the per-folder identity, not lab membership.

Detection + count

The "files you do not have permission to view" set is the cross-boundary received records, the ones carrying a received_from_fingerprint, since those were sealed and addressed to the prior identity. After takeover the new identity's openSealed() can no longer decrypt them and the new account is not the named recipient.

  • Scan users/<currentUser>/{notes,projects,methods,experiments,tasks,sequences} for records where received_from_fingerprint is present (cross-boundary inbound shares).
  • These are foreign-recipient by definition: the bundle was sealed to the prior account's X25519 key, not the taking-over account's. Count them as X. (Belt-and-suspenders: an openSealed() attempt against any retained sealed blob with the new key will throw, confirming undecryptability, but the provenance stamp alone is sufficient and cheaper.)
  • Exclusion guard: a record with no received_from_fingerprint is the user's own authored content and is never counted or removed. Intra-lab _shared_with_me.json references are also excluded.

Warning copy (Grant's words)

There are X shared files that you do not have permission to view. Taking over this folder will mean the local copies of those shared documents will be removed from this folder.

Shown as part of the S4 "Take ownership" confirm, before any re-key, with the count resolved from the detection scan. If X is 0 the line is omitted. Honors the no-soft-lock rule: the user can still pick "Use a different folder" or "Continue without sharing" instead of confirming.

Safe removal on confirm

  • On confirm only, delete the local copies of the detected records (the <id>.json plus any per-record attachment subfolder, e.g. notes/<id>/Images/). Trash-not-delete where the folder's delete convention supports it.
  • Drop any matching entries from the intra-lab manifests so no dangling references remain, but do not remove records that the manifest points at in other users' directories (those are not the prior account's sealed shares).
  • Untouched: every record with no received_from_fingerprint. The new account's own authored notes, tasks, methods, projects, experiments, and sequences are strictly preserved. The previous owner's published directory profile and their devices are also untouched (consistent with the S4 security note).
  • Re-key the folder identity sidecar (users/<username>/_sharing_identity.json, read via readSharingIdentity() in identity/sidecar.ts) to the taking-over account as the final step.

Security note on "Take ownership" (S4)

ResearchOS is local-first, so whoever holds the folder on disk and is signed in can already read and rewrite everything in it. Re-keying the folder identity to their account is consistent with that reality, not a new exposure. But it must be an explicit, clearly-worded choice, and it must NOT touch the other account's identity anywhere else.

  • Taking ownership re-keys this folder's identity sidecar to the current account. The previous owner's published directory profile and their keypair on their own devices are untouched.
  • This connects to the provider-rebind spec (2026-06-19-account-provider-rebind.html), which is the broader "change the login bound to an identity" problem. S4 is the folder-local case of it.
