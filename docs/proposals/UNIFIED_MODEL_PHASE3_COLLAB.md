# Unified Data Model, Phase 3 build scope (notes collaboration)

Status, DRAFT for Grant sign-off (2026-06-05). This is the build spec for Phase 3, real-time collaboration on a note, the last layer of the depth-first notes vertical (store = Phase 1 done, version control = Phase 2 done). Parent design is `UNIFIED_DATA_MODEL.md` (section 8, the E2E-blind relay) and `CROSS_BOUNDARY_SHARING_COLLABORATE.md` (section 11, the phasing). The collab posture and substrate are already LOCKED; this scopes the build.

## 0. Why this is different from Phase 1 and 2

Phases 1 and 2 were pure local builds, fully testable in node + jsdom, landed and verified live by Grant in one browser. Phase 3 is NOT pure local:
- It needs a RELAY (a Cloudflare Worker + Durable Object), the one new piece of infrastructure the whole unified-model effort has been pointing at. The local spikes never needed it.
- It is END-TO-END ENCRYPTED. The relay must never read note content.
- It cannot be verified by automated tests OR by the orchestrator. Proving collab needs TWO browser clients editing the same doc through a running relay. The orchestrator cannot run a blocking relay (watchdog) or drive two browsers (sandbox). Grant verifies, with two tabs.

So Phase 3 is build-then-Grant-runs-and-tests, with a real infra prerequisite. The honest framing up front.

## 1. Goal and non-goals

Goal, the locked MVP (cross-boundary-sharing section 11), TWO-PERSON live editing of a single note's text, with live cursors, through an E2E-blind relay, started from the existing Share button and retired back to local. Going live is the SAME local Loro doc connecting a relay backend (design doc section 10, backend c), not a second editing system.

Non-goals (deferred, do NOT build here):
- N-person sessions, late-join snapshot/compaction catch-up, methods/experiments/folders collab (cross-boundary-sharing section 11 "Expansion"). MVP is two-person, text, in-memory (a connected peer re-broadcasts on reconnect).
- Server-side persistence of the doc. The MVP relay holds nothing at rest; if both peers disconnect the session ends (retire-to-local captures the result first).
- Replicating to other entities. That is the post-notes-vertical arc.

## 2. Architecture, three pieces

Per design doc section 10, collab is backend (c) on the SAME store. Three pieces:

1. The RELAY (Durable Object), a blind byte-fan-out. One DO instance per note-session (room). It authenticates a signed socket, then forwards every binary message from one peer to the others without ever parsing the payload (the payload is already ciphertext). Adapt `spikes/collab-yjs/src/worker.ts` (the WebSocket-Hibernation DO with webSocketMessage/Close/Error fan-out); swap the Yjs framing for Loro. The fan-out mechanic is identical under encryption, the relay never reads the bytes.
2. The CLIENT PROVIDER (`frontend/src/lib/loro/collab/relay-provider.ts`), the network layer loro-codemirror does NOT provide. It wires the local Loro doc + EphemeralStore to the relay socket:
   - `doc.subscribeLocalUpdates(bytes => encrypt+sign(bytes); ws.send(...))`, every local change goes out.
   - on relay message, `decrypt+verify; doc.import(bytes)`, every remote change comes in.
   - `ephemeral.subscribeLocalUpdates(bytes => send)` and on receive `ephemeral.apply(bytes)`, cursors/presence (loro-codemirror's LoroEphemeralPlugin already drives the EphemeralStore from CM6 selections).
   - on connect, broadcast our state (export update from empty) so the peer catches us up; the still-connected peer re-broadcasts current state for reconnect (the MVP no-persistence fallback).
3. The CRYPTO (reuse `frontend/src/lib/sharing/encryption.ts`), E2E-blind (locked posture 4a, secsync shape). A per-doc symmetric key K is generated by the initiator and wrapped to the invited collaborator's X25519 key with the existing `sealToRecipient`. Each update is XChaCha20-Poly1305-encrypted under K and Ed25519-signed before it hits the relay; the relay verifies the signed envelope and fans the ciphertext. Reuses the X25519/Ed25519 identity keys the sharing/directory feature already ships, and gives verified-sender provenance for free (it dovetails with the verified-sender provenance already on imported notes).

## 3. The collab session lifecycle

- START, from the existing Share button on a note, a new "Collaborate live" action. The initiator generates K, opens a relay room (the DO, addressed by a random session id), wraps K to the invited collaborator via the directory + sealToRecipient, and sends a signed invite through the existing relay mailbox (the cross-boundary-sharing send path). The collaborator accepts, unwraps K, and connects to the same room.
- LIVE, both clients run the same Loro doc bound to the editor (the Phase 1 binding) PLUS the relay provider (this phase). Edits and cursors flow both ways through the blind relay. Version control (Phase 2) keeps working unchanged, every change is still a Loro commit in each client's native history, attributed via the actors map (the collaborator's peer resolves to their identity).
- RETIRE, either peer ends the session through the existing `note-transfer.ts` / import path with external-copy provenance, then the DO tears down. The offline-at-retire collaborator gets the final snapshot relayed through the existing mailbox (cross-boundary-sharing open question, recommended).

## 4. Wire protocol (verified Loro 1.12.3 API)

- Outbound doc changes, `doc.subscribeLocalUpdates((bytes: Uint8Array) => ...)` fires with the incremental update bytes on every local commit. Encrypt + sign + send.
- Inbound doc changes, `doc.import(bytes)` applies a remote update (idempotent, CRDT-merged). No conflict handling needed, Loro merges.
- Catch-up on join, `doc.export({ mode: "update", from: <peer state vector> })` sends only what the peer lacks; the simplest MVP just exports the full update from empty on connect.
- Cursors/presence, the EphemeralStore (`ephemeral.subscribeLocalUpdates` / `ephemeral.apply` / `ephemeral.encode`) carries the loro-codemirror cursor state; it is EPHEMERAL (never in the synced doc, design doc section 5) and expires on disconnect.

## 5. Infrastructure (the prerequisite only Grant can provide)

The relay is a Cloudflare Worker + Durable Object, the one new deploy target (like R2 / Neon were). Two ways to get there:
- DEVELOP against a LOCAL relay first, `wrangler dev` runs the DO locally via workerd with no Cloudflare account (exactly how the collab-yjs spike ran). Build + Grant-tests the whole thing locally (two tabs against localhost), defer the cloud deploy. The orchestrator cannot run the blocking `wrangler dev` itself (watchdog), so Grant runs it during testing.
- Or PROVISION the Cloudflare Worker + DO now (a real deploy target, env wiring like the existing R2/Resend infra), and develop against the deployed relay.

Recommend local-wrangler-first, prove the whole collab loop on localhost (no account, no spend, fast iteration), then a thin deploy step at the end. Either way Grant runs the relay + the two browser tabs for verification.

## 6. Persisted-data-shape / protocol decisions (sign-off gate)

1. The relay message envelope, a small signed binary frame `{ sessionId, senderPubKey, kind: doc|ephemeral, ciphertext }` that the DO authenticates (signature + clocks) without reading the ciphertext. Confirm the secsync-style shape.
2. The session id, a random opaque id addressing the DO room (no note id in it, so the relay cannot correlate sessions to notes). Confirm.
3. Reuse the existing identity keys (X25519/Ed25519) + the directory + sealToRecipient for K-wrapping and the invite. Confirm (no new key material).
4. The actors map (Phase 2 chunk 1) gains the collaborator's peer -> their directory identity when their first remote update arrives, so version rows attribute to the real person. Confirm.

## 7. Work breakdown (chunks, in dependency order)

1. The relay DO + worker, adapt `spikes/collab-yjs/src/worker.ts` into a real `relay/` worker (signed-socket auth, blind fan-out, WebSocket Hibernation), with a `wrangler.toml` for local dev. Testable with a tiny node ws harness (two fake clients, one fans to the other), no browser.
2. The crypto envelope, encrypt/sign/verify helpers over `encryption.ts` (per-doc K, the signed frame), unit-tested in node (round-trip + tamper-rejection).
3. The relay provider (`collab/relay-provider.ts`), the doc.subscribeLocalUpdates <-> import + ephemeral loop over the (encrypted) socket, unit-tested with two in-memory docs + a fake relay (convergence + cursor relay, the collab-yjs spike's test shape).
4. The session UX, the "Collaborate live" Share action, the invite (reuse the directory + mailbox), the connect/disconnect + live-cursor binding in the editor, retire-to-local. Surgical + flag-gated behind LORO_PILOT_ENABLED.
5. The deploy step (only if Grant provisions Cloudflare), `wrangler deploy` the relay, wire the prod URL + env. Otherwise stays local-dev.

After chunks 1-4, Grant runs `wrangler dev` + two browser tabs (two accounts / two profiles) on a scratch folder, opens the same note, and confirms live two-way editing + cursors + retire. That is the only way to verify it; jsdom and the orchestrator cannot.

## 8. Exit criteria

- Two browser clients, each editing the same note, see each other's text changes live and each other's cursor, through the relay.
- The relay never sees plaintext (it fans ciphertext; a captured frame is undecryptable without K).
- Version control keeps working in each client (changes are still Loro commits, attributed to the right person via the actors map).
- Retire-to-local ends the session and leaves each side with the merged note + external-copy provenance.
- Flag-off, zero collab surface; the note editor is exactly Phase 1+2.
