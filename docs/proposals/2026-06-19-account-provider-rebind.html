Account provider rebind
Phase 4, researcher social layer. Spec for review, not built.

Change my third-party login, keep everything else

A researcher signs in today with a Google or ORCID account. They change institutions, lose that login, or simply want a different one. Everything about their account should stay the same (the keypair identity, username, display name, all data, lab memberships, billing) and only the linked third-party login changes.

This doc shows what breaks today, the one architectural decision that makes rebind clean, and the user-facing flow. Mark each decision Approve or Disapprove, drop comment pins anywhere, then Export.

Verdict from the code trace. No rebind flow exists today, and a naive login change orphans four of the five bindings. The local keypair is the only thing that survives, because the server keys everything by a peppered hash of the OAuth email. That single keying choice is the root cause and the thing to fix.

1. The binding chain

What a user account is actually made of, and what each piece is keyed by. The fix is to cross-key the server records to the stable keypair fingerprint so the email becomes a swappable pointer.

Today: everything keyed by email hash

K
Local identity (keypair)
Ed25519 + X25519 in the sidecar. The real, durable identity. Username, name, all data hang off this.
x
Directory (email to pubkey)
Primary key is email_hash. New email is a new hash, so a second orphaned entry, identity split in two.
x
Lab membership
The OAuth email is sealed under the lab key as emailHashEnc. verifyMemberEmailBinding rejects the lab login on any mismatch.
x
Account settings + profile
Keyed by ownerKey = email hash. The @handle, avatar, E2E settings blob all point at the old hash.
x
Billing / Stripe
owner_key primary key = email hash. New login looks like a brand-new unpaid account, plan and Stripe customer orphaned.

Proposed: cross-keyed to the keypair fingerprint

K
Local identity (keypair)
Unchanged. Already the stable anchor. The fingerprint derived from it becomes the join key the server trusts.
ok
Directory
Keep an index by fingerprint (one already exists, getBindingByFingerprint). Rebind repoints the email pointer, the keypair entry stays put.
ok
Lab membership
Member marks a rebind, the head re-seals the new email hash on next lab open. Reuses the deferred-seal reconcile that already exists.
ok
Account settings + profile
Resolve owner records by fingerprint, with the email hash as a secondary lookup. Rebind updates the pointer, records stay.
ok
Billing / Stripe
Carry the Stripe customer + subscription to the new owner key (or key billing by fingerprint too). No re-subscribe, no lost plan.
stable anchor breaks on a naive login change survives the rebind

2. The user flow

Two real cases. The common one (still have the old login) and the hard one (the old login is already dead, the graduating-student problem).

Today

  1. Settings has no such optionThere is no change-login control anywhere in the app.
  2. User signs in with the new account insteadIt mints or resolves a different keypair, or binds a different hash.
  3. Dead endThe lab login is rejected (email mismatch), billing shows unpaid, the directory profile is gone. The account looks lost.

Proposed rebind, old login still available

  1. Settings to Account to Change linked loginClear entry point, explains what stays the same and what changes.
  2. Sign in with the new provider to prove controlOAuth round trip captures the verified new email. Old session already proves the current login.
  3. Confirm the keypair fingerprintShow the safety-check fingerprint so the user confirms it is the same identity moving, not a merge of two accounts.
  4. Atomic repointDirectory + settings + profile + billing pointer move to the new email hash under the same fingerprint, in one staged step with a compensating log.
  5. Lab memberships re-sealEach lab is marked rebind-pending. The head re-seals on next open, the member sees an honest waiting state, never a lockout.

Proposed rebind, old login is gone (graduating student)

  1. Recovery code is the proof of controlThe keypair, unlocked by the recovery code, is the real identity. Holding it proves the account, even with the old email dead.
  2. Sign in with the new providerCaptures the new verified email to bind.
  3. Same atomic repoint + lab re-seal as aboveThe keypair fingerprint is the through-line, so nothing about the migration depends on the dead old login.

4. Scope, risk, and what to verify

For sizing once the decisions are set.

Crypto and security review required. Rebind changes who can act as an account. The proof-of-control gate (two logins, or recovery code) is the security boundary and needs an independent pass before it ships, same bar as the directory/relay work.

No soft-locks, ever. Every state in the flow (proof pending, repoint in progress, lab re-seal pending) must have a visible way out. A half-migrated account must be recoverable, not stranded.

Atomicity is the hard part. Five systems, two of them external (Stripe, the relay). A staged migration with a compensating log and a clear in-progress marker beats trying to make it one transaction. The lab dimension is naturally async already, so lean on that.

Land before broad PI launch. Institution changes are routine for the exact audience. This is not an edge case, it is table stakes for trusting the platform with a career-long record.
