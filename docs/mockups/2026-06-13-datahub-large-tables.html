Data Hub, large tables

Review mockup, 2026-06-13. Approve or flag each change, drop comment pins, then Export and paste back to chat.

The thing being proposed

Today the Data Hub grid is a hand-edited, cell-versioned spreadsheet. Wonderful for a 200-row table, fatal at 200,000 rows (it renders every row to the DOM and holds every cell in memory). This proposal adds a second lane that trips automatically above a size threshold. Big tables become query-backed datasets that stay on your disk, compute on your own processor through a background engine, and never try to draw themselves in full. Small tables are completely unchanged. Review each change below.

Chrome cannot draw that many rows at once, and neither can any browser. For a full hand-editable view, open the file in a spreadsheet or analysis app. Here you can preview, jump to any row, filter, transform, and analyze the entire dataset without loading it all.

Per your no-soft-locks rule, the full-render path is never a dead button. It is an honest warning that explains why the browser is the wrong tool for that one operation, points at the file and at external apps, and keeps every safe operation alive. The user is informed, not blocked.

On your disk, data-shape

dataset.jsonschema, lineage, analyses, rules

Parquet stores the table column by column and compressed, so a 60 MB CSV becomes a few MB and the engine reads only the columns an analysis needs. We keep your original file exactly as imported (it is a research record, never silently rewritten) and add the compact working copy plus a small metadata sidecar. Pasted or derived data, which has no original file, gets only the Parquet copy. Everything lives on your disk, nothing sits as a multi-gigabyte blob in browser cache. This is a new on-disk shape, flagging it as a data-shape change per your rule.
