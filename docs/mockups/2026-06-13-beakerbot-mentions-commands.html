5. Before vs after: removing the "this" guessing

click to comment
Today (no @ or /)
make a chart of this
BeakerBot has to guess what "this" means from context clues in the conversation.
The context bridge reads the page state, recent messages, and the open object to infer what "this" points to. It can be wrong, especially mid-conversation or when multiple objects are visible.
summarize the results from last week's run, the qPCR one
Phrase must spell out enough detail for the bot to resolve the right experiment, often across multiple tries.
compare those two experiments for me
"Those two" requires the model to hold conversation state and map it to specific objects, which fails after a few messages.
With @ mentions and / commands
/plot fakeGFP expression table x
Unambiguous. The table ID is sent directly. BeakerBot never guesses.
The context bridge still runs in the background (so free-text still works). @ and / are an opt-in power mode for users who want zero ambiguity. They do not replace the bridge, they short-circuit it.
/summarize fakeGFP qPCR, June 6 x
One command chip, one object chip. No prose needed.
Compare fakeGFP qPCR, June 6 x and fakeGFP qPCR, May 30 x
Two explicit chips. Conversation state does not matter; each object carries its own ID.

The core value proposition: @ and / remove the "this"/"the result" guessing the context bridge currently has to do. Free-text still works (the bridge still runs), so no user is penalized for not using them. Power users and repeat-task workflows benefit most.

7. Open design questions for Grant

click to comment
Q1. Fixed command list vs extensible
The six commands above (/summarize, /plot, /cite, /digest, /setup, /draft) map 1:1 to real BeakerBot tools. Option A is keep it curated and tight: users trust a short fixed list more than a sprawling one, and each command maps to a real tool with no guessing. Option B is make it extensible: let the list grow as new composite tools land, or eventually let PIs define custom lab macros (the "reusable named workflows" item in the backlog). The mockup shows Option A. Which way?
Q2. Chip removal: inline x button vs backspace
The mockup shows a small "x" on each chip. Alternative: pressing Backspace at the start of the text input pops the last chip off (Slack / Linear pattern). A third path is to do both. The "x" is more discoverable; backspace is faster for keyboard-heavy users. The current sketch defaults to "x" only. Any preference?
Q3. Type chip colors: multi-color vs uniform sky-blue
Section 6 shows each type with its own accent color. The existing BeakerSearch palette uses a uniform sky-blue for all rows. Keeping chips uniform (all sky-blue) is simpler and avoids a new color system. Using per-type colors makes it easier to scan a mixed-type chip list at a glance. Call it.
Q4. @ picker trigger: only "@" or also "[[" like Notion
The mockup triggers the picker on "@". Some lab apps (and Notion) use "[[" to mean "link an object". Since "@" is the standard in chat UIs, the mockup defaults there. Mention it in case there is a reason to prefer the bracket pattern.

These are the four decisions that need a call before the build starts. Marking this section "Approve" means the questions have been answered (use the comment pins to leave the answers inline).
