# qPCR summary, 2026-06-13 Three runs completed today. **FakeYeast-001** showed 2.4x higher fakeGFP expression vs WT control (p = 4.2e-5). FakeYeast-002 was not significantly different (p = 0.31). Standard curves R2 > 0.99.
Reuses the existing notes editor. Edit, then save to Notes. Discard Save to Notes

Canvas replaces the read-only card with a compact "Drafted in Canvas" line in the chat. The right panel opens with the draft in the existing markdown notes editor (no new editor to build). The user fixes a word, a number, or a heading, then clicks Save to Notes. Discard throws the draft away without writing anything. The WHY: the edit-then-save IS the consent, so Approve / Reject buttons disappear for draft-write tools. Saving without editing is fine; one click if the draft is correct as-is.

Canvas replaces the read-only approve card only for draft-write tools whose output is a complete text artifact. Destructive or outward-facing tools (deletes, sends, external submissions) keep their own final confirm card. Pure analysis and read tools never open Canvas; their output is chat text with provenance chips.

4. Open design questions for Grant to decide

click to comment
Q1: docked right panel vs overlay?
Option A: docked right panel (mockup above)
Canvas lives beside the chat whenever a draft is open. Chat narrows to make room.
Pro: always visible, no layer to manage. Matches VS Code / Cursor canvas pattern.
Con: narrows the chat on smaller screens. Competes with the main app area behind the BeakerBot panel.
Option B: overlay sheet
A wide sheet slides in from the right (or up) when a draft arrives. Chat is still visible behind it.
Pro: full width for the editor, no layout reflow.
Con: hides the chat context the user just sent. Must remember to dismiss before continuing.
Q2: multiple drafts, queue as tabs or one at a time?
Option A: one at a time (simpler)
Only one draft in Canvas at once. A new draft replaces the current one (with a "save or discard first" guard).
Pro: simple, always clear what is being edited.
Con: in Auto mode, BeakerBot may fire several write tools in one plan. The first draft gets pushed out before the user reviews it.
Option B: tabs in the Canvas header
Multiple drafts queue as tabs. The user switches between them and saves each.
Pro: holds multiple drafts from a single Auto-mode plan.
Con: tabs add complexity. Easy to miss an unsaved tab and close Canvas thinking you are done.
Q3: fully replace the approve card, or keep a thin card too?
Option A: Canvas fully replaces approve card (mockup above)
The chat line only says "Drafted in Canvas." No read-only card in chat. Saving = consent.
Pro: cleaner chat. Only one path to act on a draft.
Con: if the panel is collapsed and the user misses the notice, the draft sits unacted-on with no visible fallback in chat.
Option B: thin approve card in chat too
A collapsed preview card stays in the chat with an "Open Canvas" link and a quick Approve-as-is shortcut.
Pro: nothing is lost if the panel is collapsed. Power users can Approve-as-is without opening Canvas.
Con: two paths to the same action. Risks users always approving in chat and never using Canvas.
Q4: raw markdown edit or live-preview split?
Option A: raw markdown only
Mount the existing notes editor as-is. Toolbar generates markdown syntax.
Pro: zero new editor code. Consistent with how notes look in editing mode.
Con: researchers who do not know markdown syntax may not know how to fix bold or headings without the toolbar.
Option B: Edit + Preview tabs (mockup above)
Edit tab shows raw markdown; Preview tab renders it. Toolbar (B, I, H1) inserts syntax for the user.
Pro: approachable for non-markdown users. Preview confirms the output before saving.
Con: Preview is a rendering pass on top of the existing editor. Slight extra build work.

These four questions drive the build scope. Q1 (docked vs overlay) affects layout architecture and whether the BeakerBot panel needs a split-pane mode. Q2 (tabs vs one-at-a-time) affects Canvas state management when BeakerBot fires multiple write tools inside an Auto-mode plan. Q3 (replace vs alongside) is the consent model; the mockup shows full replacement. Q4 (raw vs split-preview) determines whether the existing notes editor can drop in unchanged.
